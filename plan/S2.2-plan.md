# Plan — S2.2: Sinh roadmap cá nhân hóa từ level + mục tiêu học

- **Epic:** EPIC-2 — Onboarding & Lộ trình học (UC03, UC04)
- **Ưu tiên:** P1
- **AC nguồn:** AC-03-02 (chính). Liên quan: AC-04-01, AC-04-02, AC-04-04 (định nghĩa data contract cho S2.4); FR-04; BR09; BR11; SRS §3.3.
- **Ngày lập:** 2026-06-06

## 1. Mục tiêu & phạm vi

Sau khi user hoàn tất onboarding (S2.1 — đã lưu `target_level`, `interests`, `daily_minutes`, `words_per_session` vào `user_profile`), hệ thống phải có khả năng **sinh ra một roadmap học tập cá nhân hóa**: một danh sách topics có thứ tự, dựa trên level đã chọn + các topics user quan tâm, chỉ gồm nội dung đang active (BR11).

Phạm vi S2.2 = **logic + API sinh roadmap + data contract**. Không build màn hiển thị snake-style (đó là S2.4). "Done" = có endpoint `GET /api/roadmap` trả về cấu trúc roadmap ổn định mà S2.4 consume được, phủ AC-03-02 và cung cấp đủ field cho AC-04-01/02/04.

## 2. Acceptance Criteria cần thỏa

- [ ] **AC-03-02:** User chọn level A1–C2 hợp lệ + submit onboarding hợp lệ → hệ thống lưu level (S2.1 đã làm) **và tạo được roadmap cá nhân hóa từ active content của level đó**. S2.2 hiện thực hóa vế "tạo roadmap".
- [ ] **AC-04-01** (hỗ trợ): roadmap trả về topics khớp level đã chọn — data contract đúng để S2.4 render.
- [ ] **AC-04-02** (hỗ trợ): roadmap chỉ phản ánh content `is_active = TRUE` (Admin/crawler quản lý) — query lọc `is_active`.
- [ ] **AC-04-04** (hỗ trợ): không có topic active cho level → trả về roadmap rỗng (empty state), KHÔNG xóa level/profile của user.
- [ ] **BR09/BR11:** level thuộc {A1..C2}; content disabled không xuất hiện trong roadmap.

## 3. Thiết kế kỹ thuật

### Quyết định kiến trúc (và lý do)

**1. Roadmap được TÍNH on-the-fly (computed), KHÔNG lưu thành bảng riêng.**
- Lý do: AC-04-02 yêu cầu roadmap luôn phản ánh active content (Admin bật/tắt, crawler cập nhật hàng tuần). Nếu snapshot roadmap vào bảng `user_roadmap`, sẽ phải đồng bộ lại mỗi khi content đổi → phức tạp, dễ stale. Roadmap = phép chiếu (`topic` active theo `level` + sắp theo `interests` rồi `display_order`) nên rẻ để tính mỗi request.
- Schema hiện có (`level.display_order`, `topic.level_id`, `topic.display_order`, `topic.is_active`, `user_profile.target_level`, `user_profile.interests`) đã đủ. **Không cần migration DB mới cho S2.2.**
- Tiến độ học per-topic (completed/locked/in-progress) là phần riêng (chưa có bảng `user_topic_progress` trong schema). S2.2 trả `status` mặc định để S2.4 render snake-style; khi story tracking tiến độ được làm sau, chỉ cần bơm dữ liệu thật vào cùng field này.

**2. KHÔNG sinh roadmap bên trong `POST /api/onboarding`.**
- Lý do: vì roadmap là computed, `POST /api/onboarding` chỉ cần lưu profile (đã đúng ở S2.1). Tách `GET /api/roadmap` giúp S2.4 (và mọi lần mở lại màn roadmap) luôn lấy bản mới nhất, không phụ thuộc thời điểm onboarding. Tránh side-effect nặng trong request submit.

**3. Thuật toán sắp xếp roadmap (personalization).**
Topics của level đã chọn, sắp theo:
   1. Topics user đã chọn trong onboarding (`user_profile.interests`) lên trước — đúng tinh thần "cá nhân hóa từ level + goal/topics".
   2. Trong mỗi nhóm, theo `topic.display_order` rồi `name`.
Chỉ lấy `is_active = TRUE` (BR11/AC-04-02). Mỗi node gắn `order_index` để S2.4 vẽ snake.

### Backend (speak-buddi-be/)

**File mới `routers/roadmap.py`** — `GET /api/roadmap`:
- Guard `Depends(current_user)` (AC-04-03: Guest bị chặn — tái dùng pattern auth như `onboarding.py`).
- Đọc `user_id` từ JWT payload (`sub`).
- Lấy profile qua `user_repo.get_user_by_id` (đã trả `level`, hoặc thêm hàm lấy `interests`).
- Nếu `target_level` NULL (chưa onboarding) → 409/400 với message gợi ý hoàn tất onboarding (AC-03-01 flow).
- Validate level thuộc `VALID_LEVELS` (tái dùng hằng từ `onboarding.py` hoặc đưa vào `core`).
- Gọi repo sinh roadmap, map sang schema, trả `RoadmapOut`.

**File `repositories/roadmap_repo.py` (mới)** — `get_roadmap_topics(db, level_code, interests)`:
- SQL: JOIN `topic` + `level` WHERE `l.code = :level AND t.is_active = TRUE`.
- Trả các field: `id, name, slug, description, display_order`.
- Sắp xếp ưu tiên interests: dùng `ORDER BY (t.slug = ANY(:interests)) DESC, t.display_order, t.name` (PostgreSQL boolean sort) — interests-first.
- Trả list dict; router gán `order_index` tăng dần + `status` mặc định.

**File `schemas/roadmap.py` (mới)** — Pydantic:
```python
class RoadmapNode(BaseModel):
    id: str
    name: str
    slug: str
    description: str | None = None
    order_index: int          # 0-based, thứ tự vẽ snake
    is_interest: bool          # user đã chọn topic này khi onboarding
    status: str = "available"  # available | completed | in_progress | locked (placeholder cho tracking sau)

class RoadmapOut(BaseModel):
    level: str                 # "B1"
    level_name: str            # "Intermediate (B1)"
    total_topics: int
    nodes: list[RoadmapNode]
```

**Đăng ký router** trong `main.py` (nơi include các APIRouter) — tái dùng pattern include `onboarding.router`.

**Tái dùng/điều chỉnh `repositories/user_repo.py`:** `get_user_by_id` chưa trả `interests`. Thêm 1 hàm nhỏ `get_profile_for_roadmap(db, user_id)` trả `{target_level, interests}` (hoặc bổ sung `interests` vào `_USER_SELECT`). Ưu tiên hàm riêng để không ảnh hưởng auth/me.

### Frontend (speak-buddi/)

S2.2 **không build UI** (màn roadmap snake-style do S2.4 làm). Chỉ cần chuẩn bị tầng gọi API để S2.4 dùng ngay:
- **File mới `src/features/roadmap/services/roadmapService.js`** — `getRoadmap()` gọi `GET /api/roadmap`, đính kèm JWT (tái dùng http helper/`onboardingService.js` làm khuôn mẫu).
- Không đổi flow `OnboardingPage.jsx` (vẫn `navigate("/dashboard")` sau submit). Nếu cần, có thể trỏ sang `/roadmap` khi S2.4 sẵn sàng — quyết định để S2.4.

### Data model / DB (nếu có)

- **Không thêm bảng / cột mới.** Roadmap computed từ `level`, `topic`, `user_profile` đã tồn tại.
- Index đã đủ: `idx_topic_active_order (is_active, display_order)`, `idx_topic_level (level_id)`.
- Ghi chú TBD: bảng tracking tiến độ per-topic (`user_topic_progress`) sẽ cần ở story tracking sau — field `status` trong schema đã chừa chỗ.
