## 1. Phân tích yêu cầu

### 1.1 Model phân quyền (SRS §4.5)

SRS định nghĩa **2 chiều độc lập**:

| Chiều | Giá trị | Mô tả |
|---|---|---|
| **Role** | `student` / `admin` | Thuộc tính cố định gắn với tài khoản |
| **Subscription state** | `free` / `paid` | Trạng thái động — có `user_subscription.status='active'` + `expires_at > now()` (hoặc `expires_at IS NULL` cho Pro vĩnh viễn) |

> "Paid User là subscription state, không phải role riêng." — SRS §4.5

### 1.2 Hiện trạng code

**Backend (`main.py`):**
- `_make_access(user)` — JWT claims hiện có: `sub, email, name, type, exp` — **thiếu `role` và `is_paid`**.
- `current_user()` — Depends trả payload JWT, không enforce role.
- MOCK_USERS có `role: "student"` nhưng không có `is_paid`.
- Chưa có `require_admin` / `require_paid` Depends.

**Frontend (`src/shared/auth/`):**
- `AuthContext` — state: `token, isAuthenticated, login(), logout()` — **thiếu `user`, `isAdmin`, `isPaid`**.
- `ProtectedRoute` — chỉ check `isAuthenticated`, chưa có `AdminRoute` / `PaidRoute`.

**DB schema (`schema_core.sql`)** — đã sẵn sàng:
- `users.role CHECK (role IN ('student','admin'))`
- `user_subscription(user_id, status, expires_at)` — chưa dùng (vẫn là MOCK).

### 1.3 Scope S1.8

**Trong scope:**
- Thêm `role` + `is_paid` vào JWT claims.
- `require_admin` / `require_paid` FastAPI Depends.
- Cập nhật response `/api/auth/me`, `/login`, `/register`, `/google` trả thêm `role`, `is_paid`.
- Thêm mock users (admin + paid demo) vào MOCK_USERS.
- FE: `AuthContext` expose `user`, `isAdmin`, `isPaid`.
- FE: `AdminRoute` + `PaidRoute` components.

**Ngoài scope:**
- Migrate MOCK_USERS → DB (TODO S1.8 → S8.2).
- Admin UI, Payment flow, AI quota enforcement (S7.2–S9.x).
- Đăng xuất tất cả thiết bị / revoke session (S8.x).

def require_admin(user: dict = Depends(current_user)) -> dict:
    if user.get("role") != "admin":
        raise HTTPException(status_code=403, detail="Yêu cầu quyền Admin.")
    return user

def require_paid(user: dict = Depends(current_user)) -> dict:

    return user

### 3.2 Frontend — `speak-buddi/src/`

#### Task FE-1: Cập nhật `authService.js`

Thêm vào `src/shared/services/authService.js`:
- Constant `KEY_USER = "user_info"`
- `setUser(user)` → `localStorage.setItem(KEY_USER, JSON.stringify(user))`
- `getUser()` → `JSON.parse(localStorage.getItem(KEY_USER))` (try/catch → null)
- `clearUser()` → `localStorage.removeItem(KEY_USER)`
- Cập nhật `clearTokens()` gọi thêm `clearUser()`.

**Không** dùng JWT decode phía FE (tránh logic trùng, trust server response).

#### Task FE-2: Cập nhật `AuthContext.jsx`

```jsx
export function AuthProvider({ children }) {
  const [token, setToken] = useState(() => getToken());
  const [user, setUserState] = useState(() => getUser());

  // Đồng bộ đa tab
  useEffect(() => {
    function handleStorage(e) {
      if (e.key === KEY_ACCESS) setToken(e.newValue || null);
      if (e.key === KEY_USER)   setUserState(e.newValue ? JSON.parse(e.newValue) : null);
    }
    function handleAuthChanged() {
      setToken(getToken());
      setUserState(getUser());
    }
    window.addEventListener("storage", handleStorage);
    window.addEventListener(AUTH_CHANGED, handleAuthChanged);
    return () => { ... };
  }, []);

  function login({ access_token, refresh_token, user }) {
    setTokens({ access_token, refresh_token });
    setUser(user);              // lưu localStorage
    setToken(access_token);
    setUserState(user);
    dispatchAuthChanged();
  }

  function logout() {
    clearTokens();              // clearTokens đã gọi clearUser() bên trong
    setToken(null);
    setUserState(null);
    dispatchAuthChanged();
  }

  const value = {
    token,
    user,
    isAuthenticated: !!token,
    isAdmin: user?.role === "admin",
    isPaid:  !!user?.is_paid,
    login,
    logout,
  };

  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}
```

export default function AdminRoute() {

## 6. Rủi ro & lưu ý

| Rủi ro | Xử lý |
|---|---|
| JWT claim `is_paid` stale khi subscription expire giữa phiên | Ghi TODO: `/api/auth/me` cần re-verify; FE có thể poll hoặc check khi mount protected route |
