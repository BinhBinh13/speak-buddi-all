## 1. Tóm tắt

S1.1 là story "nền móng": repo, tech stack, cấu trúc folder, kết nối DB và base layout responsive. **Thực tế dự án đã chạy được** — S1.2/S1.3/S1.4/S1.5 đã `done`, nên FE scaffold (Vite + React 19 + Bootstrap + react-router) và auth flow đã tồn tại. Vì vậy story này **KHÔNG làm lại** scaffold mà tập trung vào phần còn thiếu của một "setup chuẩn":

1. **Chuẩn hóa cấu hình môi trường (ENV)** cho cả FE và BE (hiện BE chưa có `.env.example`, chưa có `requirements.txt`).
2. **Đặt nền tảng kết nối DB** ở mức tối thiểu, có chủ đích (config + bộ schema khởi tạo) — không build ORM/migration đầy đủ vì các P0 data-model story (S3.1, S4.1) sẽ làm chi tiết. Cần quyết định một điểm gây nhập nhằng: SRS schema-draft là **MySQL** còn story yêu cầu **PostgreSQL** (xem §6).
3. **Verify & harden base layout responsive** theo NFR §4.8: breakpoints 375/768/1024/1440, touch target ≥ 44px, keyboard/ARIA cơ bản — kiểm tra trên các page đã có (landing, pricing, login, register, dashboard, speaking).

"Done" = repo có cấu trúc rõ ràng + ENV chuẩn hóa (FE + BE) + kết nối DB ở mức nền (config + script schema chạy được hoặc được chốt là TBD có lý do) + tất cả page hiện tại đạt 4 breakpoint với touch target ≥ 44px, có tài liệu/checklist verify cho Reviewer.

## 3. Phân tích hiện trạng

### Những gì ĐÃ CÓ (KHÔNG cần làm lại)

**Frontend (`speak-buddi/`):**
- Scaffold Vite 8 + React 19 chạy được; `package.json` đủ deps: bootstrap 5.3, react-bootstrap, react-router-dom 7, react-icons, recharts.

- Base layout shell **đã có**: `src/shared/components/AppLayout.jsx` (sidebar 240px + topbar fixed + `MobileBottomNav`, có `@media (max-width:768px)` ẩn sidebar). Public shell: `PublicNavbar` + `PublicFooter`.
- Routing đã cấu hình: public (`/`, `/pricing`, `/login`, `/register`, `/oauth/callback`) + protected (`ProtectedRoute` bọc `/dashboard`, `/speaking`).
- Auth client: `shared/auth/AuthContext.jsx`, `ProtectedRoute.jsx`, `shared/services/{apiClient.js,authService.js}`.
- Design tokens FE đã tồn tại 2 nguồn: `shared/constants/theme.js` (palette emerald/navy/cream + Playfair/DM Sans — dùng cho dashboard/speaking) và `shared/constants/designTokens.js` (token theo mockup cho landing/pricing).
- `index.css` đã set `#root` full-width, nền `#FAFAF7`.
- Một số `@media` query đã rải rác ở 13 file (landing/dashboard/pricing).

**Backend (`speak-buddi-be/`):**
- FastAPI app chạy được (`main.py`): `/health`, `/speak`, `/tts`, `/api/auth/{register,login,refresh,me}`.
- Auth JWT tự ký (HMAC-SHA256) access 15p + refresh 7 ngày, validate email/password theo §5.2, không log secret/token.
- CORS đã cấu hình qua `ALLOWED_ORIGINS`. Repo Git riêng (`.git` lồng), có `.gitignore`, `README.md`.
- ENV được đọc qua `python-dotenv`: `ANTHROPIC_API_KEY`, `ELEVENLABS_API_KEY`, `ELEVENLABS_VOICE_ID`, `JWT_SECRET`, `ALLOWED_ORIGINS`. `check_env.py` kiểm 6 package + 5 key.

### 4.2 DB — kết nối + schema cốt lõi

> **Quyết định cần chốt với user (xem §6):** SRS `schema-draft.sql` viết cho **MySQL** (ENGINE=InnoDB, `ENUM`, `JSON`...), còn story ghi **PostgreSQL**. Phải chọn 1 trước khi viết schema thật. Plan đề xuất PostgreSQL (theo story) nhưng cần xác nhận; nếu giữ MySQL thì tái dùng nguyên `schema-draft.sql`.

Phạm vi DB ở S1.1 là **đặt nền**, không phải build toàn bộ data layer:
- Thêm vào BE một module kết nối DB nhẹ, ví dụ `speak-buddi-be/db.py`: đọc `DATABASE_URL`, tạo engine/connection (SQLAlchemy core hoặc driver thuần — chọn nhất quán phong cách "helper thuần" hiện có của BE), cung cấp `get_db()` dependency. KHÔNG ép buộc ORM phức tạp.
- Thêm script schema cốt lõi đặt trong repo BE, ví dụ `speak-buddi-be/db/schema_core.sql` — chỉ **nhóm Identity/Auth + Payment/Subscription** cần cho EPIC-1/EPIC-8:
  - `users` (email unique, password_hash nullable, role student/admin, status, timestamps) — khớp §5.3.
  - `user_profile` (target_level, learning_goal, timezone Asia/Ho_Chi_Minh).
  - `oauth_account`, `user_session` (refresh_token_hash, expires_at) — chuẩn bị S1.6/S1.7/S1.8.
  - `payment_plan`, `user_subscription` (status active/expired/cancelled/pending) — để xác định "Paid User là subscription state, không phải role" (§4.5, BR04).
  - Giữ index như draft: `uq_users_email`, `idx_users_role_status`, `idx_user_subscription_user_status`.
  - Soft-delete pattern (`is_active`) ở các bảng content sẽ do S3.1/S9.2 làm — S1.1 chỉ cần `payment_plan.is_active`.
- **Các bảng learning/quiz/speaking/translation/report KHÔNG tạo ở S1.1** — thuộc S3.1, S4.1, các epic tương ứng. Ghi rõ ranh giới này.
- **Auth vẫn dùng `MOCK_USERS`** trong S1.1 để không phá S1.4/S1.5; việc nối auth vào DB thật là follow-up (S1.8). Ghi chú TODO trong code.
- Nếu hạ tầng DB hosting là TBD (§4.10) và team chưa dựng được Postgres → fallback: chốt schema + config nhưng đánh dấu phần "chạy migration thật" là **TBD** trong notes, không block toàn story.

### 4.3 FE — verify responsive breakpoints, touch target ≥ 44px

- Tạo file constant breakpoint dùng chung, ví dụ `speak-buddi/src/shared/constants/breakpoints.js`:

  để các `@media` rời rạc tham chiếu nhất quán (theo §4.8 + DESIGN.md: container max-width 1280px, margin mobile 16px / desktop 64px).

  - 768px: chuyển sidebar↔bottomnav (đã có ngưỡng này — verify đúng).
  - 1024px: sidebar + content; right panel có thể ẩn/thu gọn.
  - 1440px: content container giới hạn max-width 1280px (DESIGN.md "Fixed-Fluid Hybrid"), căn giữa.
- **Audit touch target ≥ 44px**: `MobileBottomNav` item, icon-button trên `DashTopbar`/`PublicNavbar`, nút submit form `LoginPage`/`RegisterPage`, link CTA. Tăng `min-height/min-width: 44px` + padding nơi thiếu.

  - Đảm bảo `:focus-visible` thấy được trên link/nút chính.

  - Kiểm tương phản chữ/nền các màu primary/secondary đạt AA.
- **KHÔNG đổi visual design** các page đã done (landing/pricing dùng mockup) — chỉ sửa khi vi phạm breakpoint/touch target/a11y.
