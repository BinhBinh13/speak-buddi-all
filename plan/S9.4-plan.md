## 1. Mục tiêu & phạm vi

Bổ sung **xử lý lỗi** cho crawler Langeek đã có ở S9.3: khi Langeek không reachable, scrape/parser lỗi, hoặc guardrail chặn — hệ thống **giữ nguyên nội dung active hiện tại** (cache fallback), **retry** theo policy cấu hình, **ghi log chi tiết**, và **thông báo Admin** (AC-13-05, BR12, FR-20).

"Done" = mô phỏng lỗi crawl (fixture `--fail` hoặc env) → job `failed` + **không** thay đổi/xóa topic_word active; `GET /api/admin/crawler/sync-status` hiển thị `cache_active=true`, `last_success_at`, `retry_status`, lý do lỗi; retry tự động chạy lại và ghi log; Admin nhận email (SMTP cấu hình) hoặc banner trên `/admin/crawler`; học viên vẫn thấy nội dung cũ ở `/vocabulary`.

> **Phạm vi loại trừ:**
> - **Happy path crawl + auto-publish** → đã xong **S9.3**.
> - **Admin chỉnh/disable nội dung crawl** → **S9.5** (AC-13-06, BR11).
> - **Manual curation fallback khi crawl bị cấm pháp lý vĩnh viễn** — SRS TBD; S9.4 chỉ notify + gợi ý tắt crawler (`is_enabled=false`), không build import CSV.
> - **Push notification / Slack** — ngoài SRS v1.1.0; dùng email Admin + in-app banner (mockup crawler page).

### Kiến trúc luồng lỗi

```
run_sync()
  ├─ success → (S9.3 — không đổi)
  └─ failure
       ├─ KHÔNG gọi publish_batch / KHÔNG soft-disable words
       ├─ finish_job(status=failed, retry_status=...)
       ├─ append_log(error) + payload { source_url, last_success_at, cache_active, retry_count }
       ├─ schedule_retry() nếu retry_count < max_retries
       └─ notify_admin_crawler_failure() — email + flag cho FE banner
```

### Backend (speak-buddi-be/)

#### 3.1 Schema mở rộng — `db/schema_crawler.sql` (migration idempotent)

Thêm cột (ALTER IF NOT EXISTS pattern hoặc file `schema_crawler_s9_4.sql`):

| Bảng | Cột mới | Mục đích |
|---|---|---|
| `content_source` | `max_retries SMALLINT DEFAULT 3` | Policy retry |
| `content_source` | `retry_delay_seconds INTEGER DEFAULT 300` | Delay giữa các lần retry (5 phút) |
| `content_source` | `notify_email VARCHAR(255)` | Email Admin nhận cảnh báo (fallback env) |

Status job giữ `failed`; retry tạo **job mới** `trigger_type=retry` hoặc re-run cùng job — **khuyến nghị:** job mới `trigger_type='manual'|'scheduled'` với parent log link qua `payload_json.parent_job_id` để audit đơn giản.

**Retry policy (MVP — PO TBD finalize):**
- `max_retries=3`, delay cố định `retry_delay_seconds` (config qua PUT `/config`).
- Retry chạy subprocess/scrape lại; nếu vẫn fail → `retry_status=exhausted` + notify lần cuối.
- Không exponential backoff phức tạp trừ khi dư thời gian.

**Notify Admin:**
- Email tới `content_source.notify_email` hoặc env `ADMIN_CRAWLER_NOTIFY_EMAIL` hoặc query `users WHERE role='admin'` (email đầu tiên).
- Skip an toàn nếu SMTP chưa cấu hình (log warning — pattern S8.3).
- §4.5: không log failure message chứa secret; truncate reason.

**Cache fallback (correctness):**
- Failure path: **early return** trước `publish_batch`.
- Verify: query active word count trước/sau failed job **không đổi**.
- Learner APIs (`learning.py`) — không sửa; đã filter `is_active=TRUE`.
