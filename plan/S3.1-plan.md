## 1. Mục tiêu & phạm vi

### Data model / DB

#### ERD tóm tắt
```
level (1) ───< (N) topic ───< (N) topic_word >───< (N) tag
                  │                    │   (qua topic_word_tag)
                  └── level_id         └── level_id (denormalize để lọc nhanh theo level)

level.id          PK UUID
topic.level_id    FK → level.id        (ON DELETE SET NULL)
topic_word.topic_id FK → topic.id      (ON DELETE CASCADE)
topic_word.level_id FK → level.id      (ON DELETE SET NULL)  ← denormalize cho query lọc theo level
topic_word.created_by FK → users.id    (ON DELETE SET NULL)  ← admin tạo tay; NULL nếu từ crawler
topic_word_tag (topic_word_id, tag_id) PK kép — bảng nối M:N
```

#### Cột chính từng bảng (PostgreSQL, theo convention `schema_core.sql`)

#### Index
- `level`: unique `code`, unique `name`.
- `topic`: unique `name`, unique `slug`, idx `(level_id)`, idx `(is_active, display_order)` (lọc nội dung active theo thứ tự).
- `topic_word`: unique `(topic_id, word)`, idx `(topic_id, is_active)`, idx `(level_id)`, idx `(created_by)`.
- `tag`: unique `name`, unique `slug`.
- `topic_word_tag`: PK kép + idx `(tag_id)` (truy ngược từ tag → words).

#### Soft delete approach
- **Dùng `is_active BOOLEAN DEFAULT TRUE`** — bám đúng SRS §5.3 ("Soft delete via `is_active=false`") và AC-13-03, đồng bộ với `payment_plan.is_active` đã có trong `schema_core.sql`. **Không dùng `deleted_at`** để khớp policy SRS và schema hiện tại.
- Áp cho `topic` và `topic_word`. `level`, `tag`, `topic_word_tag` không có (level cố định; tag/quan hệ M:N dùng hard-delete khi cần dọn).
- Query đọc của FE/BE sau này (S3.2/S9.1) phải luôn filter `WHERE is_active = TRUE` cho người học; admin xem cả inactive.

## 5. Test & cách verify
- **Chạy schema:** `psql -U postgres -d speakbuddi -f db/schema_core.sql` rồi `psql -U postgres -d speakbuddi -f db/schema_learning.sql` → không lỗi; chạy lại lần 2 vẫn không lỗi (idempotent) — verify §5.3 tạo bảng.
- **Đủ bảng:** `\dt` thấy `level, topic, tag, topic_word, topic_word_tag` → §5.3.
- **Level A1–C2:** `SELECT code FROM level ORDER BY display_order;` trả đúng 6 mã A1..C2 → §5.3.
- **Soft delete:** `UPDATE topic SET is_active=false WHERE …; SELECT count(*) FROM topic WHERE is_active=true;` giảm đúng, row vẫn tồn tại (không bị xóa) → AC-13-03 / §5.3.
- **FK + lọc theo level:** `SELECT w.word FROM topic_word w JOIN topic t ON w.topic_id=t.id WHERE t.level_id=(SELECT id FROM level WHERE code='A1') AND w.is_active=true;` trả từ mẫu → §3.4.
- **Chống trùng:** insert lại `topic` cùng `name`/`slug` hoặc `topic_word` cùng `(topic_id, word)` → vi phạm UNIQUE (chuẩn bị AC-13-02).
- **Trigger updated_at:** `UPDATE topic SET description='x' WHERE …;` → `updated_at` đổi.
- Không cần unit test code (chưa có route). Map AC → từng query verify ở trên.
