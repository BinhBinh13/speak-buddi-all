# Plan: S1.5 — Đăng nhập + JWT access/refresh token

- **Epic:** EPIC-1 — Nền tảng & Xác thực (UC02, F02, FR-02, BR01)
- **Ưu tiên:** P0 (blocker cho S1.6 OAuth, S1.7 reset, S1.8 phân quyền)
- **AC nguồn:** AC-02-02 (UC02) · SRS §4.5 (Security: JWT access + refresh token, không log secret/token) · SRS §5.2 (Wrong login / Empty auth fields / Expired token)
- **Ngày lập:** 2026-06-06

## 1. Tóm tắt
Hoàn thiện luồng đăng nhập email/password đã có ở dạng sơ khai (`/api/auth/login` + `LoginPage`) để **đúng đặc tả**:
1. Xử lý sai thông tin theo AC-02-02 — hiển thị đúng message `⚠ Email hoặc mật khẩu không đúng.` (hiện BE trả tiếng Anh `Invalid email or password`, FE LoginPage còn dùng copy tiếng Anh).
2. Áp dụng chiến lược **JWT access token (ngắn hạn) + refresh token (dài hạn)** theo §4.5 (hiện chỉ phát 1 token 7 ngày, không tách access/refresh, không có endpoint refresh).
3. Bảo mật theo §4.5: không log password/token; tự động làm mới access token khi hết hạn bằng refresh token; xử lý phiên hết hạn (`⚠ Phiên đăng nhập hết hạn...`).

**Định nghĩa "done":** user nhập đúng → nhận access + refresh token, vào app; nhập sai → đúng message §5.2, không đăng nhập; access token hết hạn → FE tự refresh bằng refresh token (không phải đăng nhập lại); refresh token hết hạn/không hợp lệ → buộc đăng nhập lại với message phiên hết hạn. Không có secret/token nào bị log.

## 2. AC nguồn (từ SRS)
- [ ] **AC-02-02** — Sai email/password → hệ thống hiện `⚠ Email hoặc mật khẩu không đúng.` và **không** đăng nhập. (BE trả message Việt ở §5.2; FE hiển thị nguyên văn.)
- [ ] **§5.2 Wrong login** — message chính xác: `⚠ Email hoặc mật khẩu không đúng.` (BE không phân biệt "email sai" vs "password sai" để chống user-enumeration).
- [ ] **§5.2 Empty auth fields** — bỏ trống email/password → `⚠ Vui lòng điền đầy đủ thông tin.` (FE chặn trước submit; BE validate lại).
- [ ] **§5.2 Expired token** — access token hết hạn và không refresh được → redirect `/login` + `⚠ Phiên đăng nhập hết hạn. Vui lòng đăng nhập lại.`
- [ ] **§4.5 Token strategy** — phát hành JWT **access token + refresh token**; access ngắn hạn (15 phút), refresh dài hạn (7 ngày); có endpoint `/api/auth/refresh`.
- [ ] **§4.5 Security** — không log password/token; password chỉ so khớp dạng hash; dùng `hmac.compare_digest` (đã có).
- [ ] **AC-02-01 (regression)** — `/api/auth/register` cũng phải trả về cặp access+refresh token mới (đồng bộ shape với login để FE dùng chung).

## 4. Thiết kế kỹ thuật

### Backend (speak-buddi-be/main.py)

**4.1 Token helper — tách access/refresh.** Thêm hằng + hàm cạnh khối JWT hiện có (dòng 56–84):
```
ACCESS_TTL  = 15 * 60          # 15 phút (§4.5 access ngắn hạn)
REFRESH_TTL = 7 * 86400        # 7 ngày
def _make_access(user)  -> str  # _sign({sub,email,name,type:"access",  exp:+ACCESS_TTL})
def _make_refresh(user) -> str  # _sign({sub,email,         type:"refresh", exp:+REFRESH_TTL})
```
- Giữ nguyên `_sign`/`_verify` (đã kiểm `exp`). Thêm kiểm `payload["type"]` ở chỗ tiêu thụ: `current_user` chỉ chấp nhận `type == "access"`; endpoint refresh chỉ chấp nhận `type == "refresh"` → tránh dùng nhầm refresh token làm access.

**4.2 `POST /api/auth/login` (sửa).**
- Trim/normalize email lowercase (đồng bộ register đã lowercase).
- Nếu `not email or not password` → `400` `detail="⚠ Vui lòng điền đầy đủ thông tin."`
- Nếu user không tồn tại **hoặc** sai password → `401` `detail="⚠ Email hoặc mật khẩu không đúng."` (một message chung — chống user-enumeration, đúng AC-02-02).
- Thành công → trả:
  ```json
  { "access_token": "...", "refresh_token": "...", "user": { id, name, email, level, streak, goal } }
  ```
- **Logging:** chỉ `log.info("LOGIN ok user=%s", user["id"])` — **không** log email body/password/token (§4.5). Có thể log `LOGIN fail email=<masked>` nếu cần audit nhưng không log password.

**4.3 `POST /api/auth/refresh` (mới).** Đặt ngay dưới `/api/auth/login`.
- Schema `RefreshRequest(BaseModel): refresh_token: str`.
- Verify refresh token qua `_verify`; nếu `type != "refresh"` → `401`.
- Tra `MOCK_USERS` theo `email`/`sub` để dựng lại user hiện tại (đảm bảo user còn tồn tại) → phát **access token mới**. Trả `{ "access_token": "..." }`.
- Lỗi (hết hạn/sai chữ ký/không phải refresh) → `401` `detail="⚠ Phiên đăng nhập hết hạn. Vui lòng đăng nhập lại."`
- (Tùy chọn, ghi TBD) Refresh-token rotation: phát refresh mới mỗi lần refresh — bỏ qua ở giai đoạn mock (chưa có store để revoke); ghi nhận khi có DB.

**4.4 `POST /api/auth/register` (sửa nhẹ — regression).** Đổi response từ `{token, user}` sang `{access_token, refresh_token, user}` để FE dùng chung 1 luồng lưu token. (Cập nhật đồng bộ FE authService/RegisterPage — xem task.)

**4.5 `current_user` (sửa nhẹ).** Sau `_verify`, nếu `payload.get("type") not in (None, "access")` → `401`. Giữ tương thích token cũ (`type` vắng mặt) trong giai đoạn chuyển tiếp.

### Security (SRS §4.5)
- Không `log` password, access/refresh token, hay Authorization header (rà soát: các `log.info` mới chỉ chứa `user_id`).
- Password so khớp bằng hash + `hmac.compare_digest` (đã có ở login dòng 360) — giữ nguyên.
- Message login dùng **chung một câu** cho cả "email không tồn tại" và "sai password" → chống user-enumeration (đúng tinh thần AC-02-02).
- Access token ngắn hạn (15 phút) giảm thiệt hại nếu rò rỉ; refresh dài hạn (7 ngày).
- **Brute-force:** SRS không quy định rate-limit cụ thể cho login ở v1.1.0; **không bắt buộc** trong story này. Ghi nhận là cải tiến tương lai (cần DB/Redis để đếm) ở mục Rủi ro — không tự thêm phụ thuộc.
- Token lưu `localStorage` (đồng bộ AuthContext hiện hành). Ghi nhận trade-off XSS vs httpOnly cookie là TBD khi có DB/session store thật.
