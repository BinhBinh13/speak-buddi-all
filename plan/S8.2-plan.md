## 1. Mục tiêu & phạm vi

Nối tiếp luồng UC10 từ S8.1: khi **thanh toán thành công**, payment provider gọi **webhook về backend**, hệ thống **xác thực callback → đánh dấu `payment_transaction` thành `success` → tạo/kích hoạt `user_subscription` (status `active`, set `starts_at`/`expires_at` theo `payment_plan.duration_days`) → người dùng trở thành Paid User** (BR04). Webhook phải **idempotent** (xử lý trùng không kích hoạt 2 lần — dựa trên UNIQUE `(provider, provider_transaction_id)` đã có sẵn từ S8.1).

"Paid User" trong dự án là **subscription state, không phải role** (SRS §4.5, BR04; `auth/deps.py:require_paid` đọc `is_paid` từ JWT, `user_repo.get_is_paid` truy `user_subscription` status `active` + chưa hết hạn). Vì vậy S8.2 chỉ cần ghi đúng `user_subscription` — `is_paid` sẽ tự đúng ở lần login/refresh kế tiếp (xem mục 6 điểm "JWT refresh").

**"Done" =**
- Có endpoint webhook (không dùng JWT user) xác thực bằng signature/API key của provider, parse payload, map về transaction tương ứng.
- Thanh toán thành công → `payment_transaction.status='success'`, `paid_at` set, `raw_payload` lưu DB; tạo/cập nhật `user_subscription` `active`; gắn `user_subscription_id` ngược lại transaction.
- Idempotent: gọi lại cùng callback → trả 200 nhưng không tạo subscription lần 2.
- `SepayProvider` được thêm vào `services/payment_providers/` + đăng ký factory; `MockPaymentProvider.verify_callback()` được implement đầy đủ để chạy dev end-to-end.
- FE: MockPayPage nút "Giả lập thành công" gọi webhook thật (dev) thay vì chỉ điều hướng; FE có cách biết kết quả (poll trạng thái transaction) — tối thiểu, không thiết kế lại UI.

**Ngoài phạm vi (chuyển story khác):**
- Xử lý fail/cancel + retry + email + màn `thanh_toan_thanh_cong/that_bai` → **S8.3** (AC-10-03). S8.2 chỉ chạm nhánh success; nhánh failed/cancelled chỉ cập nhật status tối thiểu nếu callback báo, phần UX để S8.3.
- Đổi voice/model → **S8.4**.
- Refund (`status='refunded'`) → ngoài phạm vi cả Epic hiện tại.

### 3.0 Hiểu cơ chế Sepay (quan trọng — khác VNPay/MoMo)

Sepay là dịch vụ **đối soát chuyển khoản ngân hàng qua VietQR**, KHÔNG phải cổng redirect-checkout truyền thống:
- Người dùng quét VietQR / chuyển khoản tới **số tài khoản ngân hàng** với **nội dung chuyển khoản chứa mã thanh toán** (vd `SB <code>`).
- Khi tiền vào tài khoản, Sepay gửi **webhook HTTP POST** tới backend với payload (đã xác nhận qua docs Sepay):
  - `id` (mã giao dịch Sepay), `gateway` (tên ngân hàng), `transactionDate`, `accountNumber`, `code` (mã thanh toán Sepay tách được từ nội dung), `content` (nội dung CK đầy đủ), `transferType` (`in`/`out`), `transferAmount`, `referenceCode`, `subAccount`, `accumulated`, `description`.
  - Auth: header `Authorization: Apikey <API_KEY>` (cấu hình trong dashboard Sepay; Sepay cũng hỗ trợ HMAC-SHA256/OAuth2 — dùng Apikey cho đơn giản).
  - Backend phải trả **HTTP 200/201 + `{"success": true}`** trong **30s**, nếu không Sepay retry tối đa 7 lần (Fibonacci, trong 5 giờ) → bắt buộc idempotent.

**Hệ quả thiết kế (khác plan S8.1 đã giả định "redirect_url ra ngoài"):**
1. `create_checkout()` của SepayProvider KHÔNG trả URL trang provider để nhập thẻ. Thay vào đó sinh **mã thanh toán** nhúng vào transaction + trả `redirect_url` tới **màn thanh toán nội bộ FE hiển thị VietQR + hướng dẫn chuyển khoản** (img.vietqr.io hoặc QR của Sepay). Mã thanh toán phải **map 1-1 về transaction** để webhook đối soát.
2. **Cách map webhook → transaction:** webhook không biết `transaction.id` của ta; phải dựa vào **payment code trong nội dung CK**. Quy ước: dùng `provider_transaction_id` = mã thanh toán ta tự sinh (vd `SB<8hex>` hoặc dùng chính phần đầu `transaction.id`), nhúng vào nội dung VietQR; webhook đọc `code`/`content` → tra ngược transaction. UNIQUE `(provider='sepay', provider_transaction_id)` đảm bảo idempotency + map.
3. **Verify số tiền:** `transferAmount` từ webhook phải ≥ `payment_transaction.amount_vnd` (chống thiếu tiền). Lệch → đánh dấu nghi ngờ, không activate (ghi log, để S8.3/admin xử lý).

> GIẢ ĐỊNH cần PO xác nhận trước khi tích hợp Sepay thật (mục 6): API Key webhook, số tài khoản/subAccount nhận tiền, và quy ước prefix mã thanh toán (Sepay cấu hình regex tách `code` từ nội dung). MockProvider không cần các thông tin này → vẫn chạy end-to-end ngay.

### 3.1 Backend (`speak-buddi-be/`)

#### a) Endpoint webhook mới — `routers/payment.py` (sửa, thêm route)

```
POST /api/payment/webhook/{provider}        (provider ∈ {sepay, mock})
Auth:    KHÔNG dùng current_user (gọi từ server Sepay).
         Xác thực qua provider.verify_callback(): header Authorization "Apikey <key>"
         (Sepay) hoặc field result (mock). Sai → 401/403.
Request: JSON payload thô của provider (Sepay payload / mock payload).
Response: 200 { "success": true }  (luôn 200 khi đã nhận hợp lệ, kể cả idempotent skip)
Errors:
  401/403  signature/api key sai
  400      payload không parse được / thiếu mã thanh toán
  404      không tìm thấy transaction khớp (ghi log, vẫn cân nhắc trả 200 để Sepay
           ngừng retry nếu chắc chắn không phải giao dịch của hệ thống — xem mục 6)
```

- Nhận `Request` thô để lấy cả header (`Authorization`) + body JSON → truyền vào `provider.verify_callback(payload, headers)`.
- Pattern: thêm route vào `routers/payment.py` hiện có (cùng `prefix="/api"`, `tags=["payment"]`, logger `speakbuddi.payment`).

```python
class SepayProvider(PaymentProvider):
    name = "sepay"
    create_checkout(transaction, plan, user) -> CheckoutResult:
        # provider_ref = mã thanh toán tự sinh (vd "SB" + tx.id[:8] upper) — nhúng nội dung CK
        # redirect_url = {FRONTEND_URL}/payment/sepay?tx=<id>  (màn FE hiển thị VietQR + hướng dẫn)
        #   (hoặc trả thẳng URL QR Sepay nếu dùng API tạo QR; tối thiểu: màn nội bộ render VietQR)
    verify_callback(payload, headers) -> CallbackResult:
        # 1. Xác thực: headers["authorization"] == f"Apikey {SEPAY_WEBHOOK_API_KEY}" → else raise (401)
        # 2. Bỏ qua transferType != "in" (chỉ tiền vào)
        # 3. Tách mã thanh toán từ payload["code"] (Sepay đã tách) hoặc regex từ payload["content"]
        # 4. provider_ref = mã thanh toán; amount_vnd = int(payload["transferAmount"])
        # 5. return CallbackResult(provider_ref, status="success", amount_vnd, raw_payload=payload)
```

- `get_active_subscription(db, user_id) -> dict | None`.
- `activate_subscription(db, user_id, plan_id, payment_ref, duration_days) -> dict`:
  - Tạo `user_subscription` mới với `status='active'`, `starts_at=NOW()`, `expires_at = NOW() + duration_days days` (nếu `duration_days=0`/lifetime → `expires_at=NULL`), `payment_ref=provider_transaction_id`.

#### g) Service — `services/payment_service.py` (sửa, thêm hàm `handle_webhook`)

```python
async def handle_webhook(db, provider_name, payload, headers) -> dict:
    provider = get_provider_by_name(provider_name)        # factory theo path param, không chỉ env
    result = provider.verify_callback(payload, headers)    # raise 401/403 nếu sai chữ ký
    tx = await payment_repo.get_transaction_by_provider_ref(db, provider.name, result.provider_ref)
    if tx is None:
        log.warning("WEBHOOK_NO_MATCH provider=%s ref=%s", provider.name, result.provider_ref)
        # trả 200 hoặc 404 tùy quyết định (mục 6) — KHÔNG log payload
        ...
    # IDEMPOTENCY: nếu tx.status == 'success' → đã xử lý → return {"success": True} (no-op)
    if tx["status"] == "success":
        return {"success": True}
    if result.status == "success":
        # verify amount
        if result.amount_vnd is not None and result.amount_vnd < tx["amount_vnd"]:
            log.warning("WEBHOOK_AMOUNT_MISMATCH tx=%s", tx["id"]); ...  # không activate, S8.3/admin
        plan = await payment_repo.get_plan_by_id(db, tx["plan_id"])
        sub = await subscription_repo.activate_subscription(db, tx["user_id"], tx["plan_id"],
                                                            result.provider_ref, plan["duration_days"])
        await payment_repo.mark_transaction_success(db, tx["id"], paid_at=NOW,
                                                    raw_payload=result.raw_payload, subscription_id=sub["id"])
        log.info("PAYMENT_SUCCESS user=%s plan=%s tx=%s amount=%d sub=%s",
                 tx["user_id"], tx["plan_id"], tx["id"], tx["amount_vnd"], sub["id"])  # KHÔNG log payload
    else:
        await payment_repo.mark_transaction_failed(db, tx["id"], result.status, result.raw_payload)  # tối thiểu
    return {"success": True}
```

- **Idempotency 2 lớp:** (1) check `tx.status == 'success'` trước khi activate; (2) UNIQUE `(provider, provider_transaction_id)` ở DB chống tạo transaction trùng (đã có) + activate trong cùng transaction DB (`async with db.begin()` / commit cuối) để tránh race khi Sepay retry song song.
- **Factory cần biến thể theo tên** (path param `{provider}`), không chỉ đọc env: thêm `get_provider_by_name(name)` vào `__init__.py` (env-based `get_provider()` giữ cho checkout).

S8.2 implement đúng theo mô hình Sepay; phần màn QR đầy đủ có thể đẩy sang S8.3 nếu cần — webhook + activation (trọng tâm AC-10-02) làm trọn ở S8.2. **Cần PO xác nhận** dùng Sepay tự render VietQR hay dùng API tạo QR của Sepay. |
| 2 | **Credential Sepay (PO cấp)** | `SEPAY_WEBHOOK_API_KEY`, số tài khoản/`subAccount` nhận tiền, prefix mã thanh toán (Sepay cấu hình regex tách `code` từ nội dung CK). Chưa có → vẫn code & test bằng MockProvider end-to-end; chỉ chặn bước "test với Sepay thật". **Không blocker cho việc bắt đầu implement.** |
| 3 | **Idempotency & race khi Sepay retry** | Sepay retry tối đa 7 lần nếu không nhận 200 trong 30s. Phải: (a) check `tx.status=='success'` trước activate; (b) dựa UNIQUE `(provider, provider_transaction_id)`; (c) thực hiện trong 1 DB transaction (commit cuối). Cần đảm bảo `db/connection.py` commit đúng (kiểm pattern các router khác). |
| 4 | **JWT `is_paid` không tự cập nhật realtime** | `is_paid` nhúng trong access token lúc login/refresh (`auth/jwt.py`, `routers/auth.py`). Sau khi webhook activate, JWT cũ vẫn `is_paid=false` tới khi token refresh (ACCESS_TTL=15') hoặc login lại. **Chấp nhận ở S8.2** (đúng kiến trúc hiện có). Cải tiến tùy chọn: sau thanh toán thành công, FE chủ động gọi `/api/auth/refresh` để lấy token mới `is_paid=true`. Ghi note để S8.3 (màn success) gọi refresh. |
| 5 | **Webhook 404 vs 200 khi không khớp transaction** | Nếu trả 404, Sepay retry 7 lần. Với giao dịch không thuộc hệ thống (người chuyển sai nội dung) nên trả 200 để dừng retry + log để admin đối soát thủ công. **Cần PO/dev chốt** chính sách; mặc định plan đề xuất: log warning + trả 200 (tránh retry vô ích) nhưng KHÔNG activate. |
| 6 | **Verify số tiền & gia hạn** | Lệch tiền (thiếu) → không activate. Trùng plan đang active → S8.2 tạo bản ghi active mới là đủ cho `get_is_paid`; logic cộng dồn `expires_at` (gia hạn) đề xuất làm gọn ở S8.2 nếu rẻ, nếu không note cho story gia hạn sau. |
| 7 | **DB thật chưa chắc chạy trong dev** | Như S8.1: BE tối giản, có thể chưa kết nối Postgres. Code bám `AsyncSession`; verify thật cần dựng DB + chạy `schema_core.sql` + `schema_payment.sql`. Không cần migration mới ở S8.2. |
| 8 | **§4.5 không log payload/secret** | `raw_payload` chỉ lưu JSONB; tuyệt đối không log payload/`Authorization`/API key. Khi log lỗi verify dùng `type(exc).__name__`. |

**Phụ thuộc:** S8.1 (done) cung cấp schema + interface + checkout. Không bị chặn bởi story chưa làm. PO đã chốt provider = Sepay → đủ điều kiện code; chỉ chờ credential để test với Sepay thật.
