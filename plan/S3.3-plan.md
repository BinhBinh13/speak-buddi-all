- **Epic:** EPIC-3 — Học từ vựng (UC05 · F05 · FR-05)
- **Ưu tiên:** P1
- **AC nguồn:** AC-05-03 (chính); liên quan AC-05-04 (S3.4 — phải bảo toàn progress khi crawl, S3.3 đặt nền tảng)
- **Ngày lập:** 2026-06-06

## 1. Mục tiêu & phạm vi

Cho phép Student lưu lại tiến độ học từng từ khi học flashcard: đánh dấu một từ là **đã thuộc** (`known`) hoặc **cần luyện thêm** (`learning`), kèm ngữ cảnh **level + topic** (AC-05-03 yêu cầu "updates word/topic progress with level and topic context"). Tiến độ được persist trong PostgreSQL, gắn theo `user_id`, và topic-level progress được suy ra (derived) từ trạng thái các từ.

"Done" = nút **Đã thuộc**/**Cần luyện thêm** trong màn học từ vựng gọi API lưu thật, reload lại topic sẽ thấy trạng thái cũ; progress bar phản ánh số từ đã `known` (không chỉ vị trí flashcard hiện tại); endpoint GET trả progress của topic để FE hiển thị.

**Trong phạm vi S3.3:**
- Bảng `user_word_progress` (per-word: known/learning) + index/FK.
- Endpoint upsert progress 1 từ; endpoint GET progress theo topic (per-word + tổng hợp topic).
- FE: nối 2 nút action vào API, hydrate trạng thái khi mở topic, progress bar dựa trên số từ đã thuộc.

**Ngoài phạm vi (không làm ở đây):**
- Spaced repetition / scheduling (review interval) — chỉ lưu trạng thái + đếm lần xem.
- Streak/gamification, roadmap progress aggregate cấp level (để EPIC-2/4 dùng).
- Bảo toàn progress khi crawl đổi nội dung — đó là AC-05-04 / S3.4 (S3.3 chỉ thiết kế schema sao cho S3.4 khả thi: FK trỏ tới `topic_word.id` ổn định, không xoá row khi từ bị soft-delete).

## 3. Thiết kế kỹ thuật

### Data model / DB (file mới: `speak-buddi-be/db/schema_progress.sql`)

Tạo file riêng theo pattern hiện có (core → learning → progress), idempotent. Chạy **sau** `schema_core.sql` và `schema_learning.sql`.

Bảng `user_word_progress` (per-word, là nguồn sự thật; topic progress là derived qua aggregation — tránh trùng lặp dữ liệu):

```sql
CREATE TABLE IF NOT EXISTS user_word_progress (
    id            UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id       UUID        NOT NULL REFERENCES users (id) ON DELETE CASCADE,
    topic_word_id UUID        NOT NULL REFERENCES topic_word (id) ON DELETE CASCADE,
    topic_id      UUID        NOT NULL REFERENCES topic (id) ON DELETE CASCADE,   -- snapshot context (AC-05-03)
    level_id      UUID        REFERENCES level (id) ON DELETE SET NULL,            -- snapshot level context (AC-05-03)
    status        VARCHAR(20) NOT NULL DEFAULT 'learning'
                              CHECK (status IN ('known', 'learning')),
    review_count  INTEGER     NOT NULL DEFAULT 0,        -- số lần đánh dấu (analytics nhẹ)
    last_seen_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, topic_word_id)     -- 1 từ / 1 user → 1 dòng (upsert key)
);

CREATE INDEX IF NOT EXISTS idx_uwp_user_topic   ON user_word_progress (user_id, topic_id);
CREATE INDEX IF NOT EXISTS idx_uwp_user_level   ON user_word_progress (user_id, level_id);
CREATE INDEX IF NOT EXISTS idx_uwp_user_status  ON user_word_progress (user_id, status);
```

- Trigger `set_updated_at()` cho bảng này (dùng lại hàm từ schema_core, đăng ký theo pattern DO-block ở 2 file kia).
- **Không** dùng `ON DELETE CASCADE` thì sẽ mất progress khi admin xoá từ — nhưng `topic_word` dùng **soft-delete** (`is_active`), hiếm khi hard-delete; CASCADE chỉ kích hoạt khi hard-delete thật. Chấp nhận được, và phù hợp S3.4 (soft-delete giữ row → giữ progress).
- `topic_id`/`level_id` lưu snapshot tại thời điểm học để query theo ngữ cảnh không phải JOIN ngược nhiều bảng (thoả "with level and topic context").

> Cập nhật phần header comment của `schema_core.sql`/`schema_learning.sql` (thứ tự chạy) để thêm bước 3: `schema_progress.sql`. Tuỳ chọn — không bắt buộc, ghi rõ trong README/log.

### Backend (`speak-buddi-be/`)

**Pydantic schemas — `schemas/learning.py` (bổ sung) hoặc file mới `schemas/progress.py`:**
- `WordProgressUpsert` (request body): `{ status: Literal['known','learning'] }` — `topic_word_id` lấy từ path.
- `WordProgressOut`: `{ topic_word_id, topic_id, level_id, status, review_count, last_seen_at }`.
- `TopicProgressOut`: `{ topic_id, total_words, known_count, learning_count, percent_known, words: list[WordProgressOut] }`.

**Helper resolve user_id (QUAN TRỌNG — xem mục 6):**
- JWT `sub` = `u_001` (mock string), **không phải UUID** trong bảng `users`. Cần một helper `resolve_db_user_id(payload, db)` đặt ở `auth/deps.py` hoặc `routers/learning.py`:
  - Lấy `email = payload["email"]`.
  - `INSERT ... ON CONFLICT (LOWER(email)) DO UPDATE` (hoặc `SELECT` rồi `INSERT` nếu chưa có) vào bảng `users` để có UUID thật, trả về `users.id`.
  - Đây là "lazy provisioning": mock auth vẫn dùng được, nhưng progress FK trỏ tới UUID hợp lệ.

**Endpoints (thêm vào `routers/learning.py`, prefix `/api` sẵn có):**

1. `PUT /api/topics/{topic_id}/words/{word_id}/progress`
   - Body: `WordProgressUpsert`.
   - Verify `word_id` thuộc `topic_id` và `topic_word.is_active = TRUE` (404 nếu không).
   - Resolve `db_user_id`; lấy `level_id` từ `topic_word`.
   - `INSERT ... ON CONFLICT (user_id, topic_word_id) DO UPDATE SET status=..., review_count = user_word_progress.review_count + 1, last_seen_at = NOW()`.
   - Trả `WordProgressOut`.

2. `GET /api/topics/{topic_id}/progress`
   - Resolve `db_user_id`.
   - Trả `TopicProgressOut`: aggregate count known/learning trên các từ active của topic + danh sách per-word progress (để FE hydrate trạng thái từng flashcard).
   - 404 nếu topic không tồn tại/inactive (đồng nhất với `get_topic_words`).

> Giữ phong cách hiện có: raw SQL qua `text()` + `db.execute`, `Depends(current_user)` + `Depends(get_db)`, `response_model`, `log.info` không log token.
