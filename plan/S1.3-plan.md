# Plan — S1.3: Chặn Guest truy cập tính năng học → redirect login

- **Epic:** EPIC-1 — Nền tảng & Xác thực
- **Ưu tiên:** P1
- **AC nguồn:** AC-01-03 (UC01), BR01; liên quan AC-04-03, FR-02, SRS §4.5 (JWT access + refresh token)
- **Ngày lập:** 2026-06-06

## 1. Mục tiêu & phạm vi

Thêm một lớp **route guard ở Frontend** để mọi route tính năng học (dashboard, speaking/AI, vocabulary, quiz, translation, pronunciation, roadmap, analytics, settings...) chỉ render khi người dùng đã đăng nhập. Guest (chưa có JWT hợp lệ trong `localStorage`) khi truy cập route bảo vệ phải bị **redirect về `/login`** và **không render nội dung bảo vệ** (không flash UI). Sau khi login thành công thì quay lại đúng route đã yêu cầu (qua `?next=`).

Phạm vi story này là **logic/guard phía FE** — không có UI mockup, không tạo trang login/register thật (S1.4/S1.5). "Done" = Guest không vào được bất kỳ route học nào; user có token vào bình thường; có cơ chế giữ-và-trả-về URL đích sau login.

## 3. Thiết kế kỹ thuật

### Cơ chế guard (tóm tắt)
Dùng **component wrapper `<ProtectedRoute>`** lồng trong cây route của `react-router-dom` v7 (KHÔNG dùng route-level loader vì app đang dùng `<BrowserRouter>` + `<Routes>` khai báo, không phải data router). Wrapper kiểm tra auth state qua **AuthContext**; nếu chưa auth → render `<Navigate to="/login?next=..." replace />` (redirect ngay khi render, không bao giờ render children). Đây là lựa chọn chuẩn, dễ test, dễ bọc nhiều route bằng một `<Route element={<ProtectedRoute/>}>` cha + `<Outlet/>`.

### Phân tích auth state hiện tại
- JWT lưu ở `localStorage` key `"token"` (set tại `LoginPage` dòng 197 và `OAuthCallbackPage` dòng 13).
- Helper sẵn có trong `speak-buddi/src/shared/services/authService.js`: `getToken()`, `isAuthenticated()` (chỉ check `!!token`), `logout()`.
- `apiClient.js` đã có xử lý 401 → xóa token + `window.location.href = "/login"` (hard redirect).
- **Chưa có AuthContext / `useAuth` / AuthProvider** nào. Hiện chỉ đọc `localStorage` trực tiếp rải rác.
- **Quyết định:** Tạo mới `AuthContext` + `useAuth` mỏng, bọc giá trị từ `authService` (single source = `localStorage`), để guard và navbar dùng chung phản ứng nhất quán. Không phá vỡ `authService` (giữ làm lớp truy cập token cấp thấp).

## 7. Component design — `<ProtectedRoute>`

```jsx
// src/shared/auth/ProtectedRoute.jsx
import { Navigate, Outlet, useLocation } from "react-router-dom";
import { useAuth } from "./AuthContext";

export default function ProtectedRoute() {
  const { isAuthenticated } = useAuth();
  const location = useLocation();

  if (!isAuthenticated) {
    const next = encodeURIComponent(location.pathname + location.search);
    return <Navigate to={`/login?next=${next}`} replace />;
  }
  return <Outlet />;
}
```

- **Props:** không có (đọc auth từ context); dùng làm `element` của `<Route>` cha bọc các route con qua `<Outlet/>`. Có thể bổ sung biến thể nhận `children` để bọc lẻ 1 route nếu cần.
- **Logic:** chưa auth → `<Navigate replace>` (replace để Back không kẹt vòng lặp) kèm `?next=` = path+search hiện tại. Đã auth → `<Outlet/>` render route con.
- **Render:** khi chưa auth KHÔNG render children → đảm bảo "không render nội dung bảo vệ" (AC-01-03), tránh flash UI vì `<Navigate>` chuyển hướng ngay trong render pass.

`AuthContext` (phác thảo):
```jsx
// đọc token từ authService.getToken(); expose { isAuthenticated, token, login, logout }
// login(t): localStorage.setItem("token", t) + setState + dispatchEvent("auth-changed")
// logout(): localStorage.removeItem("token") + setState(null) + dispatch + navigate("/login")
// lắng nghe window "storage" + "auth-changed" để re-render guard/navbar
```

## 8. UX redirect flow

1. Guest mở `/speaking` → `ProtectedRoute` thấy `!isAuthenticated` → `Navigate("/login?next=%2Fspeaking", replace)`.
2. Tại `/login`, sau khi login thành công → đọc `next` từ `URLSearchParams`, `navigate(decodeURIComponent(next) || "/dashboard", { replace: true })`.
3. Giữ `?next=`: **CÓ** — bảo toàn deep link, UX tốt. Whitelist: chỉ chấp nhận `next` bắt đầu bằng `/` (same-origin path) để tránh open-redirect; nếu không hợp lệ → fallback `/dashboard`.
4. Nếu user đã đăng nhập mà vào `/login` → (tùy chọn nâng cấp) redirect `/dashboard`; không bắt buộc cho AC này.

## 9. Edge cases

- **Không có token:** guard chặn → `/login`. (Case chính)
- **Token giả/hỏng định dạng:** guard FE chỉ check tồn tại nên vẫn cho qua; lớp bảo vệ thực sự là `apiClient` — call API đầu tiên trả 401 → xóa token + về `/login`. Chấp nhận cho v1 (defense-in-depth ở BE).
- **Token hết hạn:** giống trên — 401 từ BE → `apiClient` redirect. (Story này không tự verify exp; refresh-token flow thuộc S1.5/F02.)
- **Refresh token:** SRS §4.5 yêu cầu access + refresh token; **TBD** — chưa implement trong story này, ghi nhận để S1.5 bổ sung; guard hiện chỉ dựa access token tồn tại.
- **Đồng bộ logout đa tab:** `storage` event + `auth-changed` event làm guard re-evaluate → tab khác cũng bị đẩy về login khi token bị xóa.
- **Open-redirect qua `next`:** whitelist same-origin path như §8.3.
- **Flash nội dung bảo vệ:** tránh được do `<Navigate>` chạy trong render; không dùng `useEffect` để redirect (sẽ flash 1 frame).
