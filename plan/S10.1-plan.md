# Plan — S10.1: CRUD payment plan + validation (giá > 0, ≥1 feature)

- **Epic:** EPIC-10 — Admin: Gói thanh toán (UC14, F16, FR-16) · owner Dev C
- **Ưu tiên:** P1
- **AC nguồn:** AC-14-01, AC-14-02; FR-16; BR07; SRS §5.2 (validation), §5.3 (entities — `payment_plan`, soft delete via `is_active`)
- **Ngày lập:** 2026-06-07

## 1. Mục tiêu & phạm vi

Cung cấp khu vực Admin để **tạo / xem / sửa** gói thanh toán (`payment_plan`). Khi Admin submit form với name, price, duration, features hợp lệ → hệ thống **lưu** gói (AC-14-01). Khi giá không hợp lệ (≤ 0 với gói trả phí, hoặc âm), thiếu field bắt buộc, hoặc không có feature nào → hiển thị **lỗi validation inline ngay tại field** (AC-14-02), theo phong cách message §5.2 (prefix `⚠`).

"Done" = Admin (role `admin`) đăng nhập, vào màn `/admin/payment-plans`, CRUD (Create / Read-List / Update) gói thanh toán qua API thật trên PostgreSQL, với validation cả FE (inline) lẫn BE (nguồn sự thật). Validation cụ thể: **name bắt buộc**, **price ≥ 0 (gói trả phí phải > 0)**, **duration_days hợp lệ**, **≥ 1 feature**. Student/guest bị chặn khỏi route + API.

> **Phạm vi loại trừ rõ ràng (để lại cho S10.2 — `todo`):**
> - **Soft delete / disable gói (`is_active=false`) + xác nhận → AC-14-03 → S10.2.** Story này giữ sẵn cột `is_active` (đã có trong schema), list API mặc định trả tất cả để Admin thấy nhưng **không** làm nút disable/confirm. Để code S10.1 không phải refactor khi làm S10.2, endpoint Update có thể nhận `is_active` trong body (optional) nhưng UI disable chi tiết + modal confirm là việc của S10.2.
> - Trang pricing public cho user (S1.2) và flow thanh toán (Epic 8) **không** thuộc story này.

### Data model / DB

Dùng nguyên bảng `payment_plan` đã có trong `schema_core.sql` — **không sửa schema**. Cột thực tế (PostgreSQL):

| Cột | Kiểu | Ghi chú dùng trong S10.1 |
|---|---|---|
| `id` | UUID PK | trả `id::text` ra FE |
| `name` | VARCHAR(255) NOT NULL | bắt buộc |
| `price_vnd` | INTEGER, CHECK ≥ 0 | giá theo VND (số nguyên, không phải DECIMAL `price`) |
| `duration_days` | INTEGER, CHECK ≥ 0 | 0 = vĩnh viễn/free, > 0 = có hạn |
| `description` | TEXT NULL | optional |
| `features` | TEXT[] | mảng feature; cần ≥ 1 phần tử |
| `is_active` | BOOLEAN DEFAULT TRUE | giữ sẵn cho S10.2 (soft delete) |
| `sort_order` | SMALLINT DEFAULT 0 | thứ tự hiển thị card |
| `created_at` / `updated_at` | TIMESTAMPTZ | repo set `updated_at = NOW()` khi update |

> ⚠️ **Lưu ý lệch tên cột:** `schema-draft.sql` (bản MySQL tham chiếu trong docs) dùng `price DECIMAL`, `currency`, `code`, `is_free`, `features JSON`. Bản **thực tế đang chạy** (`schema_core.sql`, PostgreSQL) khác: `price_vnd INTEGER`, `features TEXT[]`, **không có** `code`/`currency`/`is_free`. **Plan này bám schema thực tế đang chạy** (`schema_core.sql`). Không tự thêm cột `code`/`currency` để tránh phải migrate.

**Soft delete:** chỉ giữ cột `is_active`, list trả cả active/inactive cho Admin. Logic disable + confirm = S10.2.

### Backend (speak-buddi-be/)

Theo đúng pattern hiện có (router `APIRouter(prefix="/api")` → repo raw SQL `sqlalchemy.text()` → Pydantic schema), tham chiếu `routers/translate.py` + `repositories/translate_repo.py` + `schemas/translate.py`.

**File mới:**
1. `speak-buddi-be/schemas/payment_plan.py` — Pydantic:
   - `PaymentPlanCreate`: `name: str`, `price_vnd: int`, `duration_days: int = 0`, `description: str | None = None`, `features: list[str]`, `sort_order: int = 0`. Validators (Pydantic `field_validator`):
     - `name`: strip, không rỗng → else raise → 422/400 với message VN.
     - `price_vnd`: `>= 0`.
     - `features`: lọc phần tử rỗng/whitespace; sau lọc phải `len >= 1`.
     - `duration_days`: `>= 0`.
   - `PaymentPlanUpdate`: như Create nhưng các field optional + cho phép `is_active: bool | None = None` (để S10.2 tái dùng; S10.1 không expose nút này trên UI).
   - `PaymentPlanOut`: `id: str`, `name`, `price_vnd`, `duration_days`, `description`, `features: list[str]`, `is_active`, `sort_order`, `created_at`, `updated_at`.
2. `speak-buddi-be/repositories/payment_plan_repo.py` — raw SQL async:
   - `list_plans(db, include_inactive=True) -> list[dict]` — `SELECT ... ORDER BY sort_order, created_at`.
   - `get_plan(db, plan_id) -> dict | None`.
   - `create_plan(db, data) -> dict` — `INSERT ... RETURNING ...` (features bind dạng list → PostgreSQL `TEXT[]`).
   - `update_plan(db, plan_id, data) -> dict | None` — `UPDATE ... SET ..., updated_at = NOW() WHERE id = :id RETURNING ...` (chỉ set field được gửi).
3. `speak-buddi-be/routers/payment_plan.py` — `APIRouter(prefix="/api/admin", tags=["admin-payment-plan"])`, mọi route `Depends(require_admin)`:

| Method | Path | Mô tả | Response |
|---|---|---|---|
| GET | `/api/admin/payment-plans` | list (include inactive cho Admin) | `list[PaymentPlanOut]` |
| GET | `/api/admin/payment-plans/{id}` | chi tiết (cho form edit) | `PaymentPlanOut` / 404 |
| POST | `/api/admin/payment-plans` | tạo gói (validate) | 201 `PaymentPlanOut` |
| PUT | `/api/admin/payment-plans/{id}` | sửa gói (validate) | `PaymentPlanOut` / 404 |

4. Wiring trong `speak-buddi-be/main.py`: `from routers import payment_plan` + `app.include_router(payment_plan.router)` (thêm dòng cạnh các `include_router` hiện có).

**Validation BE (nguồn sự thật cho AC-14-02) — message §5.2 style (`⚠`):**
- `name` rỗng → `⚠ Vui lòng nhập tên gói.`
- `price_vnd` âm → `⚠ Giá gói không được nhỏ hơn 0.`
- (nuance giá > 0) Nếu là gói trả phí (Admin nhập giá > 0 dự kiến nhưng để 0) → xem nuance dưới.
- `features` rỗng / 0 phần tử hợp lệ → `⚠ Gói phải có ít nhất 1 tính năng.`
- `duration_days` âm → `⚠ Thời hạn không hợp lệ.`
- Cách trả lỗi: ưu tiên **400 với `detail` là chuỗi message** (khớp `apiClient` đọc `err.detail`), kèm field gây lỗi để FE map inline. Đề xuất `detail` dạng `{ "field": "features", "message": "⚠ Gói phải có ít nhất 1 tính năng." }` HOẶC giữ đơn giản: 400 + `detail` chuỗi + FE đã validate inline trước (BE là lớp chặn cuối). Chọn 1 cách và ghi rõ trong implement log.

> **Nuance "giá > 0":** AC backlog ghi "giá > 0", nhưng nghiệp vụ thực tế có **gói Miễn phí giá = 0** (đã seed `'Miễn phí', 0`). Đề xuất: **BR cứng = `price_vnd >= 0`** (không âm). Quy tắc "> 0" áp dụng **có điều kiện**: nếu Admin chọn loại gói "Trả phí" (hoặc duration_days > 0 và không tick "gói miễn phí") thì bắt `price_vnd > 0` với message `⚠ Giá gói trả phí phải lớn hơn 0.`. → Cần xác nhận với user (xem §6). Mặc định plan: cho phép 0, chỉ chặn âm + bắt ≥1 feature, để không phá gói Free đã seed.
