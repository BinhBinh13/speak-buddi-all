# Plan — S12.2: Người dùng yêu cầu xóa dữ liệu cá nhân

- **Epic:** EPIC-12 — Compliance & Support (SRS §4.6, §4.7)
- **Ưu tiên:** P2
- **AC nguồn:** SRS §4.7 (Compliance Requirements — "Users may request deletion of personal data") + §4.6 (Privacy and Data Classification — phạm vi dữ liệu cần xóa) + §4.5 (Security — không log password/token) + §5.2 (Validation) + §4.8/§4.9 (UI/accessibility, tiếng Việt)
- **Ngày lập:** 2026-06-07

## 1. Mục tiêu & phạm vi

Cho phép **người dùng đã đăng nhập** (Student, không phải Admin) **tự yêu cầu xóa dữ liệu cá nhân** trực tiếp trong ứng dụng — thay thế quy trình tạm thời qua email đã mô tả trong Privacy Policy (S12.1). Luồng gồm: mục **"Vùng nguy hiểm"** trên trang Hồ sơ (`/profile`) → modal xác nhận (có bước chống xóa nhầm) → API backend xóa/anonymize dữ liệu theo phân loại §4.6 → đăng xuất client và chuyển về trang công khai.

"Done" = user đăng nhập vào `/profile`, thực hiện xóa tài khoản/dữ liệu thành công; sau đó không đăng nhập lại được bằng email cũ (tài khoản `status=deleted`, PII đã xóa/anonymize); dữ liệu học tập cá nhân (tiến độ từ, quiz attempt, lịch sử dịch, OAuth link, session, voice preference) không còn truy vấn được; Privacy Policy có thể cập nhật nhẹ trỏ tới luồng in-app (tùy chọn trong story).

**Phạm vi:** FE (Profile) + BE (endpoint + repository transaction). **Không** làm form liên hệ email hỗ trợ (S12.3). **Không** xóa bảng nội dung học chung (topic/vocabulary/test). **Không** bắt buộc CMS/Admin duyệt thủ công cho v1.1.0.

## 3. Thiết kế kỹ thuật

### Backend (speak-buddi-be/)

**Quyết định kiến trúc:** **Soft-delete + purge PII + giữ siêu dữ liệu thanh toán** (không hard-delete row `users`):
- Schema đã có `users.status IN ('active','suspended','deleted')` và unique index trên `LOWER(email)`.
- Hard `DELETE users` sẽ CASCADE mất `payment_transaction` → ảnh hưởng analytics doanh thu (S11.x). → **Giữ row user anonymized**, xóa dữ liệu học tập/cá nhân con.
- `payment_transaction` / `user_subscription` giữ lại gắn `user_id` đã anonymize (metadata giao dịch §4.6 — phục vụ kế toán); set subscription `cancelled` nếu đang `active`.

**Endpoint mới** — mở rộng `routers/profile.py`:

```
DELETE /api/profile/account
Body: { "confirm_text": "XOA", "password": "<optional>" }
Auth: Depends(current_user)
Response: 204 No Content (hoặc 200 { message })
Errors:
  400 — confirm_text sai / password sai / validation §5.2
  403 — role=admin
  404 — user không tồn tại hoặc đã deleted
```

**Schema Pydantic** — thêm vào `schemas/profile.py`:
- `DeleteAccountRequest`: `confirm_text: str`, `password: str | None = None`
- (Response rỗng hoặc `DeleteAccountOut` với message tiếng Việt)

**Repository** — thêm `repositories/user_repo.py`:

```python
async def delete_user_personal_data(db, user_id: str) -> bool
```

Thực hiện trong **một transaction** (`await db.commit()` cuối):

| Bước | SQL / hành động |
|---|---|
| 1 | `DELETE FROM translation_history WHERE user_id = …` |
| 2 | `DELETE FROM user_word_progress WHERE user_id = …` |
| 3 | `DELETE FROM quiz_attempt WHERE user_id = …` (CASCADE answers) |
| 4 | `DELETE FROM oauth_account WHERE user_id = …` |
| 5 | `DELETE FROM user_session WHERE user_id = …` |
| 6 | `DELETE FROM password_reset_token WHERE user_id = …` |
| 7 | `DELETE FROM user_voice_preference WHERE user_id = …` |
| 8 | `UPDATE user_subscription SET status='cancelled', updated_at=NOW() WHERE user_id=… AND status='active'` |
| 9 | `UPDATE user_profile SET name=NULL, target_level=NULL, learning_goal=NULL, interests=NULL, avatar_url=NULL, updated_at=NOW()` |
| 10 | `UPDATE users SET status='deleted', email='deleted+'||id||'@speakbuddi.invalid', password_hash=NULL, updated_at=NOW()` |

Pattern email anonymize: `deleted+<uuid>@speakbuddi.invalid` — tránh trùng unique index, không tiết lộ email gốc.

**Xác thực mật khẩu** (router): nếu user có `password_hash`, so sánh HMAC-SHA256 giống `routers/auth.py` login; OAuth-only (`password_hash IS NULL`) bỏ qua bước password.

**Tương lai (ngoài scope, ghi chú):** bảng pronunciation audio / AI transcript khi Epic 6/7 có schema — bổ sung DELETE tương ứng vào cùng hàm repository.

## 5. Test & cách verify

- **Thủ công — happy path:** đăng ký user test → học vài từ / làm quiz / dịch 1 câu → vào `/profile` → Xóa tài khoản → nhập `XOA` (+ password nếu có) → 204 → redirect landing, đã logout.
- **Thủ công — login blocked:** thử đăng nhập lại email cũ → fail (user deleted).
- **Thủ công — DB:** query xác nhận `users.status='deleted'`, email anonymized; `user_word_progress`, `translation_history`, `quiz_attempt` count = 0 cho user_id; `payment_transaction` vẫn còn (nếu user từng thanh toán test).
- **Thủ công — admin:** đăng nhập admin → gọi API/UI → 403.
- **Thủ công — validation:** confirm_text sai → 400; password sai → 400; không hiện lỗi chỉ bằng màu.
- **Lint:** `cd speak-buddi && npx eslint src/features/profile/**` + `cd speak-buddi-be && python -m py_compile routers/profile.py repositories/user_repo.py schemas/profile.py`.
