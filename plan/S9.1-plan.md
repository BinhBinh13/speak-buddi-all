## 1. Mục tiêu & phạm vi

Cung cấp khu vực **Admin** để CRUD ba loại nội dung học: **Topic (chủ đề)**, **Vocabulary (topic_word)** và **Vocabulary Test** (kèm question/answer). Form hợp lệ → lưu vào DB (AC-13-01); dữ liệu sai (trùng tên topic / trùng từ trong topic / thiếu trường bắt buộc) → hiển thị **lỗi validation inline ngay tại field** (AC-13-02), message đúng chuẩn §5.2. Backend là nguồn sự thật cho ràng buộc chống trùng; chỉ user `role=admin` truy cập được route + API (BR07).

"Done" = Admin đăng nhập (mock user `admin@speakbuddi.com` đã có sẵn từ S1.8), vào `/admin/*`, CRUD (Create / Read / Update — **không** soft delete) topic / word / test qua API thật trên DB PostgreSQL, validation inline FE + chặn trùng BE; guest/student bị 403 (API) và redirect (FE).

> **Phạm vi loại trừ rõ ràng:**
> - **Soft delete `is_active=false` (AC-13-03) → S9.2.** Story này chỉ Create/Read/Update. Cột `is_active` đã có sẵn trong schema; repo `soft_delete_test` đã có skeleton (để S9.2 wire). List API mặc định chỉ trả `is_active=true`.
> - **Crawler Langeek (AC-13-04/05) → S9.3/S9.4.**
> - **Làm bài quiz / chấm điểm phía học viên → Epic 4** (đã done — S9.1 chỉ là CRUD nội dung test cho Admin).

### Backend (speak-buddi-be/)

Đã có sẵn (tái dùng — KHÔNG làm lại):
- `require_admin`, `get_db`, JWT `role`, mock admin user.
- `quiz_repo.create_test / soft_delete_test / create_question_with_answers / get_test_by_id / get_questions_by_test`.

S9.1 phải bổ sung:

1. **Router admin** — file mới `routers/admin_content.py`:
   `APIRouter(prefix="/api/admin", tags=["admin-content"], dependencies=[Depends(require_admin)])`, `include_router` trong `main.py`. Toàn bộ route bên dưới đều qua `require_admin` (BR07).
2. **Repository content** — file mới `repositories/content_repo.py` cho topic/word/tag (pattern bám `quiz_repo.py` + SQL trong `learning.py`):
   - `list_topics(search, level_code, include_inactive)`, `get_topic_by_id`, `create_topic`, `update_topic`, `topic_name_exists / slug_exists (exclude_id)`.
   - `list_words(search, topic_id, level_id)`, `get_word_by_id`, `create_word` (+ sync `topic_word_tag`), `update_word`, `word_exists_in_topic(topic_id, word, exclude_id)`.
   - Tận dụng `UNIQUE` constraint của DB; bắt `IntegrityError` để trả message thân thiện (xem validation).
3. **Repository test (mở rộng `quiz_repo.py`)** — thêm `update_test`, `update_question_with_answers` / `replace_questions_for_test` (sync nested khi PUT editor), `delete_question`/`delete_answer` (phục vụ editor xoá câu). Tận dụng `create_test`, `create_question_with_answers`, `get_questions_by_test` đã có.
4. **Pydantic schemas** — bổ sung vào `schemas/learning.py` / `schemas/quiz.py`:
   - `TopicUpdate`, `TopicWordUpdate` (nếu chưa có); `VocabularyTestUpdate`.
   - `TestEditorIn` (nested): `title`, `topic_id?`, `level_id?`, `description?`, `questions: list[QuestionEditorIn]`; `QuestionEditorIn` (`question_text`, `question_type`, `display_order`, `answers: list[AnswerEditorIn]`); `AnswerEditorIn` (`answer_text`, `is_correct`, `display_order`).
5. **Validation BE (nguồn sự thật cho AC-13-02)** — helper thuần trong router/repo:
   - Required rỗng (sau strip) → `400 {"detail": "Vui lòng điền đầy đủ thông tin."}` (Pydantic `min_length=1` + check thủ công cho field text).
   - Trùng tên/slug topic → `409 {"detail": "Tên chủ đề đã tồn tại."}` (kiểm tra trước insert + bắt `IntegrityError` của `uq_topic_name`/`uq_topic_slug`).
   - Trùng từ trong topic → `409 {"detail": "Từ này đã tồn tại trong chủ đề."}` (UNIQUE(topic_id, word)).
   - Test phải có ≥1 question; question `multiple_choice` phải có ≥1 đáp án `is_correct=true` → `400` với message rõ (khớp badge "Missing Correct Answer" trong mockup). `flashcard`/`fill_blank` không yêu cầu MC answers.
   - `question_type` phải thuộc CHECK của schema (`flashcard|multiple_choice|fill_blank|grammar_mapping`).

Rủi ro còn lại (mức thấp/trung):
- **Sync nested question/answer khi PUT editor** là phần phức tạp nhất: cần xử lý add/update/delete diff (FE gửi full danh sách, BE replace hoặc upsert + xoá phần thiếu). Đề xuất MVP: trong 1 transaction, xoá toàn bộ question/answer cũ của test rồi insert lại theo payload (đơn giản, an toàn) — chấp nhận đổi id question. Nếu cần giữ id (vd để analytics) thì làm diff — ghi nhận TBD, ưu tiên replace cho MVP.
- **Map lỗi 409 → đúng field FE:** BE trả `detail` dạng câu tiếng Việt; FE map theo nội dung (`includes("chủ đề")`/`includes("Từ này")`). Để chắc chắn, có thể thêm field `field` vào body lỗi 409 (vd `{"detail": "...", "field": "name"}`) — quyết định khi implement; nếu không, dựa vào message string.
- **`mock admin` dùng DB hay RAM:** auth hiện dùng MOCK_USERS RAM (S1.8), nhưng content CRUD ghi DB thật. `created_by` = `sub` của mock admin — phải đảm bảo UUID `sub` đó tồn tại trong bảng `users` (seed) để FK `created_by` không vi phạm; nếu `created_by` nullable thì an toàn. Kiểm tra khi implement: nếu FK NOT NULL → seed admin row vào `users` hoặc cho phép NULL.
- **Mockup field vs schema:** mockup Topics card hiển thị "Lessons" (chưa có khái niệm lesson trong schema) → map "Lessons" sang word_count hoặc ẩn; "Difficulty" trong Vocabulary = `level.code`. Giữ đúng layout, thay nội dung field bằng dữ liệu schema thật.
