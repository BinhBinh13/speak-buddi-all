# Plan — S2.3: Cập nhật level trong profile → roadmap tương lai dùng level mới

- **Epic:** EPIC-2 — Onboarding & Lộ trình học (UC03, UC04 · F03, F04 · BR09)
- **Ưu tiên:** P2
- **AC nguồn:** AC-03-04, BR09 · FR-03 (phần "/profile"), SRS §UC03 Assumption ("The Student may update the selected level in profile/onboarding settings; subsequent roadmap, topic, vocabulary, and activity recommendations must use the latest selected level.")
- **Ngày lập:** 2026-06-06

## 1. Mục tiêu & phạm vi

Cho phép Student đã onboard cập nhật trình độ CEFR (A1–C2) của mình trong trang Hồ sơ/Cài đặt. Khi level mới được lưu thành công, mọi gợi ý tương lai (roadmap, topic, vocabulary, activity) phải dùng level mới. Vì roadmap (S2.2) đã được tính on-the-fly từ `target_level` mỗi lần gọi `GET /api/roadmap`, story này **chỉ cần ghi `target_level` mới vào `user_profile`** và làm tươi user state ở FE — không phải sửa logic roadmap.

"Done" = user mở trang Cài đặt, đổi level qua selector tên thân thiện, xác nhận, BE validate BR09 và lưu, FE cập nhật user state; lần sau mở roadmap (hoặc gọi lại) thấy nội dung theo level mới.

## 2. Acceptance Criteria cần thỏa

- [ ] **AC-03-04:** Khi Student cập nhật level trong profile/onboarding settings và lưu, thì roadmap/topic/vocabulary/activity recommendations tương lai dùng level mới. → BE persist `target_level`; roadmap computed on-the-fly tự phản ánh; FE refresh user state.
- [ ] **BR09:** Level mới phải thuộc {A1, A2, B1, B2, C1, C2}; giá trị ngoài tập → BE trả 400, FE chặn submit.

## 3. Thiết kế kỹ thuật

### Backend (speak-buddi-be/)

Chọn cách **thêm endpoint riêng `PATCH /api/profile/level`** (không reuse `POST /api/onboarding`) vì:
- `POST /api/onboarding` yêu cầu cả `daily_minutes` + `words_per_session > 0` và **ghi đè `interests`** (xem `update_onboarding`). Đổi mỗi level mà phải gửi lại toàn bộ payload sẽ rủi ro reset `interests` về `[]`.
- BR09 chỉ nói level mới phải hợp lệ — **không** yêu cầu reset interests; vậy đổi level phải **giữ nguyên `interests`** đã chọn.

Files:

- **`schemas/profile.py`** (mới):
  ```python
  class UpdateLevelRequest(BaseModel):
      level: str
  class UpdateLevelOut(BaseModel):
      level: str          # target_level đã lưu (đã chuẩn hóa upper)
  ```
- **`repositories/user_repo.py`** — thêm `update_level(db, user_id, level) -> dict | None`:
  - `UPDATE user_profile SET target_level = :level, updated_at = NOW() WHERE user_id = CAST(:uid AS UUID) RETURNING target_level`
  - **Chỉ động tới cột `target_level`** — `interests`, `daily_minutes`, `words_per_session` giữ nguyên.
  - `await db.commit()`; trả `dict(row)` hoặc `None` nếu không có profile.
- **`routers/profile.py`** (mới), prefix `/api/profile`, tag `profile`:
  - `PATCH /level`, guard `Depends(current_user)`.
  - Tái dùng hằng `VALID_LEVELS = {"A1","A2","B1","B2","C1","C2"}` (giống `onboarding.py`/`roadmap.py`).
  - Chuẩn hóa `level = req.level.strip().upper()`; nếu không thuộc `VALID_LEVELS` → `HTTPException(400, "⚠ Vui lòng chọn trình độ tiếng Anh hợp lệ.")`.
  - Nếu user chưa onboard (`target_level` đang NULL) thì cũng cho phép set; nhưng story này hướng tới user đã onboard.
  - Gọi `user_repo.update_level`; `None` → `HTTPException(404, "Người dùng không tồn tại.")`.
  - `log.info("PROFILE update_level user=%s level=%s", user_id, level)` — không log dữ liệu cá nhân khác (SRS §4.5).
- **`main.py`** — `from routers import ... profile` và `app.include_router(profile.router)`.

Lưu ý quyết định:
- **Không validate level mới ≠ level cũ** — đổi về cùng giá trị là no-op vô hại, không cần chặn (idempotent).
- **Không reset `interests`** khi đổi level (BR09 không yêu cầu; SRS Assumption chỉ nói recommendations dùng latest level). Roadmap repo (S2.2) tự lọc topic theo level mới; interest nào không thuộc level mới sẽ tự nhiên không xuất hiện trong roadmap mới — không cần xử lý thêm ở đây. Ghi rõ điểm này ở mục 6 để Reviewer xác nhận.

### Frontend (speak-buddi/)

UI mockup tham chiếu: `speak-buddi-docs/ui/ho_so_cai_dat_desktop/` (`code.html` + `screen.png`). Design tokens: `speak-buddi-docs/ui/speak_buddi/DESIGN.md`. Stack thật = React + Bootstrap, port từ Tailwind mockup.

Phạm vi FE S2.3 = **section "Trình độ học" trong trang Cài đặt** (tương ứng card "Current Level" / "Learning Snapshot" của mockup). Trang Cài đặt hiện **chưa tồn tại** ở FE (route `/profile`, `/settings/*` đang comment trong `App.jsx`). Tạo tối thiểu để chứa được phần đổi level; các section khác (avatar, bio, đổi mật khẩu, notifications, 2FA, delete account) **ngoài phạm vi S2.3** — render placeholder hoặc bỏ qua, ghi chú TODO.

Files dự kiến:

- **`src/features/profile/services/profileService.js`** (mới):
  ```js
  export const updateLevel = (level) =>
    apiClient("/api/profile/level", { method: "PATCH", body: JSON.stringify({ level }) });
  ```
- **`src/features/profile/components/LevelSelector.jsx`** (mới): select/dropdown 6 level với tên thân thiện, **tái dùng đúng map của `LevelStep.jsx`** (A1="Mới bắt đầu", A2="Cơ bản", B1="Trung cấp", B2="Khá", C1="Nâng cao", C2="Thành thạo") + badge màu `BADGE_COLORS`. Cân nhắc trích `LEVELS`/`BADGE_COLORS` ra `src/features/onboarding/constants` (hoặc `shared/constants`) để cả LevelStep và LevelSelector dùng chung, tránh trùng lặp.
- **`src/features/profile/ProfilePage.jsx`** (mới): layout dùng `AppLayout` (đã có shared) với topbar/sidebar; section "Thông tin học tập" hiển thị level hiện tại (`user.level`) + selector + nút "Lưu thay đổi".
  - Confirm flow: khi user chọn level **khác** level hiện tại và bấm Lưu → mở confirm dialog (react-bootstrap `Modal`) cảnh báo "Lộ trình học sẽ được cập nhật theo trình độ mới." → xác nhận mới gọi API.
  - Sau khi API trả `{ level }`: cập nhật user trong AuthContext. AuthContext hiện chưa expose setter cho user → **thêm action `updateUser(partial)`** vào `AuthContext.jsx` (merge vào user state + `setUser` localStorage + `dispatchAuthChanged`). Hoặc đơn giản gọi lại `GET /api/auth/me` rồi `updateUser`. Hiển thị toast/alert thành công.
  - Disable nút Lưu khi level chọn === level hiện tại hoặc đang submit.
- **`src/shared/auth/AuthContext.jsx`** — thêm `updateUser(partial)` vào context value (dùng cho cả các story profile sau).
- **`src/app/App.jsx`** — bật route `<Route path="/profile" element={<ProfilePage />} />` (và/hoặc `/settings`) trong nhóm `ProtectedRoute`.
- Điều hướng vào trang: link "Settings" trong `DashSidebar.jsx` (mockup có tab Settings active) — trỏ tới `/profile`.

Responsive theo SRS §4.8 (375/768/1024/1440), touch target ≥ 44px cho selector/nút.
