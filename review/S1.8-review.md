## Phân tích code chi tiết

### Backend (`speak-buddi-be/main.py`)

**Tốt:**
- MOCK_USERS, OAUTH_ACCOUNTS đã xóa hoàn toàn — không còn data cứng.
- Password reset token migrate sang bảng `password_reset_token` (hash SHA-256, one-time use, expires_at).
- `_USER_SELECT` SQL fragment tái sử dụng tốt, giảm duplication.
- `_check_db(db)` → 503 rõ ràng khi DATABASE_URL chưa set.
- `_make_access` / `_user_response` nhận `is_paid` là tham số riêng — tách biệt khỏi DB query.
- Google OAuth: 3 nhánh (login / link / tạo mới) giữ nguyên logic AC-02-03, đúng với thiết kế.
- `_db_create_user`: INSERT users + INSERT user_profile trong cùng 1 transaction → atomic.
- `_db_link_oauth`: `ON CONFLICT DO NOTHING` → idempotent, an toàn khi gọi lại.
- `login` endpoint check `status == "suspended"` → 403 (bảo mật đúng).
- `_db_get_reset_token`: hash SHA-256 không lưu raw token, đúng §4.5.
- `reset_password`: timezone-aware comparison với `datetime.now(timezone.utc)`.

**Lưu ý nhỏ (không block PASS):**
- `_db_get_user_by_id("")` với empty string sẽ raise PostgreSQL `invalid input syntax for type uuid`. Xảy ra nếu JWT payload không có claim `sub` (token malformed). Thực tế không xảy ra vì `_verify()` đã reject token lỗi trước đó, nhưng có thể thêm guard `if not user_id: return None` để defensive.
- `streak` không có trong DB schema → luôn trả 0. Cần thêm column `streak` ở story S3.x (onboarding/learning progress).

### Frontend (`speak-buddi/src/`)

**Tốt:**
- `authService.js`: `KEY_USER = "user_info"`, `getUser()` với try/catch, `clearTokens()` gọi `clearUser()` → clean.
- `AuthContext`: đồng bộ đa tab qua `"storage"` event cho cả `token` lẫn `user_info`.
- `AdminRoute` / `PaidRoute`: pattern đúng với `ProtectedRoute`, nhất quán.
- `App.jsx`: AdminRoute/PaidRoute làm placeholder — không có route con thật, không gây lỗi.
- Callers `login()`: LoginPage, RegisterPage, OAuthCallbackPage đã cập nhật truyền `user: data.user`.

**Lưu ý nhỏ (không block PASS):**
- `OAuthCallbackPage` dùng flow cũ (`?token=` query param) với `user: null` — khi user login qua callback này, `AuthContext.user` sẽ là null dù có token. Đây là flow legacy không còn active (Google OAuth giờ dùng GSI trực tiếp qua LoginPage). Có thể cleanup sau.

## Kết luận

Story S1.8 đạt yêu cầu. Toàn bộ phân quyền role/subscription đã được implement đúng spec SRS §4.5:
- **Backend:** DB-backed auth (PostgreSQL), JWT có `role` + `is_paid`, `require_admin` / `require_paid` Depends sẵn sàng.
- **Frontend:** `AuthContext` expose đầy đủ, route guards hoạt động đúng.
- **Bảo mật:** Không log sensitive data, chống user-enumeration, token hash SHA-256, suspended check.

Hai lưu ý nhỏ không block nhưng nên xử lý ở story sau:
1. Guard empty `user_id` trong `_db_get_user_by_id` (BE defensive coding).
2. `streak` column cần thêm vào DB schema ở S3.x.
