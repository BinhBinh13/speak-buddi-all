## Bugfix Review (apiClient.js dòng 57)

Bugfix thủ công: dòng 57 đổi thành `if (res.status === 401 && !shouldSkipRefresh(endpoint))`.

**Kết luận: đúng và cần thiết.**

Luồng kiểm chứng:
- Dòng 39: 401 đầu tiên + endpoint không thuộc `NO_REFRESH_PATHS` (`/api/auth/refresh`, `/api/auth/login`) → thử `refreshAccessToken()` 1 lần, retry. Fail → clear + `auth_msg` + redirect + `return`.
- Dòng 57 (sau bugfix): chỉ chạy khi vẫn 401 **và** endpoint không phải auth path.

Vì sao bugfix đúng: khi `/api/auth/login` trả 401 (sai credentials), `shouldSkipRefresh("/api/auth/login")` = `true`, nên cả block dòng 39 lẫn block dòng 57 đều bị bỏ qua. Request rơi xuống dòng 63-66 và `throw new Error(err.detail || ...)` → `LoginPage.handleEmailLogin` catch và hiển thị `⚠ Email hoặc mật khẩu không đúng.`. **Nếu thiếu** `!shouldSkipRefresh(endpoint)` ở dòng 57, 401 từ login sẽ kích hoạt `clearTokens()` + `window.location.href = "/login"` → reload trang login, nuốt mất message lỗi và gây cảm giác "redirect loop"/mất thông báo. Bugfix loại bỏ đúng hành vi sai này.

## Security Review (§4.5)

- **Không log secret:** rà các `log.info` mới — `LOGIN ok user=%s` (`:402`), `REFRESH ok user=%s` (`:446`), `REGISTER new user id=%s` (`:363`) chỉ chứa `user_id`. Không có email body / password / token / Authorization header trong log. ✅
- **Hash + compare_digest:** password so khớp `hmac.compare_digest(pw_hash, user["password_hash"])` với SHA-256 hash (`:393, :396`). ✅ (Ghi chú: SHA-256 không phải password-hashing function chuyên dụng như bcrypt/argon2 — xem Quan sát thêm; ngoài phạm vi story mock.)
- **Phân tách token type:** `current_user` từ chối token có `type != "access"` (`:110-112`); `/api/auth/refresh` từ chối token `type != "refresh"` (`:430`). Refresh token không dùng được làm access và ngược lại. ✅
- **Chống user-enumeration:** 1 message chung cho login sai. ✅
- **Tương thích ngược:** `current_user` chấp nhận token cũ không có `type` (`type is None`) — hợp lý cho giai đoạn chuyển tiếp, không phải lỗ hổng vì vẫn verify chữ ký + exp.

### 🟡 Nên sửa (không chặn PASS)
- `apiClient.js:57-61` — block "vẫn 401 sau retry" gọi `clearTokens()` + redirect `/login` **nhưng không set** `sessionStorage.auth_msg`. Edge case: refresh thành công nhưng request retry vẫn trả 401 → user bị đá ra login mà không thấy message "Phiên đăng nhập hết hạn...". Nên set `auth_msg` ở đây cho nhất quán với block dòng 47-49. Hiếm gặp, không chặn PASS.

- Không có refresh-token rotation/revoke ở giai đoạn mock — TBD khi có store, đã ghi rõ trong plan/implement.
