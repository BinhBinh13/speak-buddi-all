1. **Hết dead code:** `ConversationPage` nay import và dùng `sendMessage` từ `conversationService.js`; hai hàm trùng lặp `callSpeak()` và `getApiUrl()` đã bị xóa. Service `sendMessage` được bổ sung param `context` để hỗ trợ greeting mode.
2. **Hết anti-pattern:** `handleSend()` không còn gọi `processResult` bên trong `setMessages(prev => ...)`. Nay tính `next = [...messages, userMsg]` trước, gọi `setMessages(next)` / `setInput("")` / `setErrorBanner(null)`, rồi gọi `processResult(payload, ...)` bên ngoài updater — đúng React pattern, an toàn StrictMode.

Sau refactor, toàn bộ AC gốc S7.1 vẫn nguyên vẹn (BE không đổi; FE chỉ tái cấu trúc đường gọi API, không đổi logic 3 nhánh response / greeting / retry / degrade). Lint `npx eslint src/features/conversation/ --max-warnings 0` exit 0, sạch.

## Xác nhận 2 fix
### Fix 1 — Dead code / trùng lặp (✅ đã sửa đúng)
- `conversationService.js:27` — `sendMessage({ text, context = null, topic = null, history = [] })` có param `context`; body `...(context && { context })` (dòng 32). Greeting truyền `context:"GREETING_MODE:"` qua chính `sendMessage` (page:175-181).
- `ConversationPage.jsx:23` — `import { sendMessage } from "./services/conversationService"`; `processResult` gọi `sendMessage(payload)` (dòng 134).
- Đã xóa `callSpeak()` và `getApiUrl()`. Grep `callSpeak|getApiUrl` trong feature chỉ còn 1 match là **comment** (dòng 129 "Xử lý kết quả từ callSpeak..."), không phải code thực thi → vô hại (cosmetic, có thể cập nhật wording sau).

### Fix 2 — Anti-pattern state updater (✅ đã sửa đúng)
- `handleSend` (dòng 186-206): tính `const next = [...messages, userMsg]` → `setMessages(next)` → `setInput("")` → `setErrorBanner(null)` → `processResult(payload, { currentMessages: next })` **ngoài** updater. Không còn side-effect trong `setMessages(prev => ...)`. `buildHistory(next)` dùng đúng mảng đã có message user mới.
