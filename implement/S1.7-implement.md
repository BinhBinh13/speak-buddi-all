## Cách test thủ công

### Happy path (cần SMTP cấu hình):
1. Điền `SMTP_USER`, `SMTP_PASS` (Gmail App Password) vào `speak-buddi-be/.env`
2. Đăng ký hoặc dùng `demo@speakbuddi.com`
3. Vào `/forgot-password`, nhập email → click "Gửi yêu cầu" → thấy success box
4. Kiểm tra hộp thư, click link reset trong email
5. Nhập mật khẩu mới + xác nhận → "Cập nhật mật khẩu" → redirect `/login` với thông báo thành công
6. Đăng nhập bằng mật khẩu mới → thành công

### Dev mode (không có SMTP):
1. Để `SMTP_USER`, `SMTP_PASS` trống trong `.env`
2. POST `/api/auth/forgot-password` với email hợp lệ → BE trả 200, log WARNING (không crash)
3. Lấy token trực tiếp từ log/debug hoặc xem `PASSWORD_RESET_TOKENS` trong RAM
4. POST `/api/auth/reset-password` với token + new_password hợp lệ → trả 200

### Edge cases:
- `/reset-password` không có `?token=` → auto redirect `/forgot-password`
- Nhập 2 ô password không khớp → client-side error, không gọi API
- Dùng token đã hết hạn (>30 phút) → error + link gửi lại
- Email không tồn tại → vẫn nhận 200 cùng message (chống enumeration)

## Ngoài phạm vi / nợ kỹ thuật / TODO

- **S1.8 migration:** Khi có DB thật, thay `PASSWORD_RESET_TOKENS` dict → bảng `password_reset_token` đã chuẩn bị trong `schema_core.sql`. Bảng lưu `token_hash` (SHA-256) thay vì raw token.
- **Rate limiting:** Chưa giới hạn số lần gọi `forgot-password` per IP/email. Nên thêm khi production.
- **Lint errors pre-existing:** 5 lỗi trong `RoadmapSection.jsx` (`no-unused-vars`, `react-hooks/rules-of-hooks`) tồn tại trước S1.7 — không nằm trong phạm vi story này.
- **Token store không persist:** Khi BE restart, mọi token đang pending bị xóa. Acceptable cho dev; S1.8 fix bằng DB.
