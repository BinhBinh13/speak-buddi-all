## Đối chiếu Acceptance Criteria

- [x] **AC-10-02** — "Given payment succeeds, when webhook/callback is received, then the system activates the subscription and marks the user as Paid User."
  - Webhook `POST /api/payment/webhook/{provider}` nhận callback → `provider.verify_callback()` xác thực + parse → map về `payment_transaction` qua `(provider, provider_transaction_id)` → set `status='success'` + `paid_at` → `activate_subscription()` tạo/gia hạn `user_subscription` `status='active'` → `get_is_paid()` (đã có từ trước, đọc đúng cột này) trả `True` → user là Paid User ở lần login/refresh JWT kế tiếp.
- [x] **BR04** — "Successful payment activates Student as Paid User according to purchased plan."
  - `activate_subscription(user_id, plan_id=tx["plan_id"], duration_days=plan["duration_days"])`: `expires_at = starts_at + duration_days` (hoặc `NULL` nếu `duration_days <= 0` — lifetime); `plan_id` đúng gói đã mua (lấy snapshot từ `payment_transaction.plan_id`, không phải gói hiện tại của hệ thống).
- [x] **Idempotency** — Webhook trùng (cùng `provider_transaction_id`) → trả 200 `{"success": true}` nhưng KHÔNG tạo `user_subscription`/`payment_transaction` lần 2.
  - Lớp 1: `handle_webhook` check `tx["status"] == "success"` → no-op trả 200 ngay (log `WEBHOOK_DUPLICATE_SKIP`).
  - Lớp 2: UNIQUE `(provider, provider_transaction_id)` ở DB (đã có từ S8.1, `uq_payment_provider_tx`) chống map sai/tạo trùng.
  - Toàn bộ thao tác chạy trong 1 DB session/transaction (`get_db()` commit cuối request — `db/connection.py`), tránh race khi Sepay retry song song.
- [x] **Bảo mật webhook (§4.5)** — Endpoint KHÔNG dùng `Depends(current_user)`/JWT user; xác thực qua `provider.verify_callback(payload, headers)`:
  - `SepayProvider`: so khớp tuyệt đối header `Authorization: Apikey <SEPAY_WEBHOOK_API_KEY>` → sai/chưa cấu hình → `PermissionError` → router trả 401. KHÔNG log giá trị header/API key (chỉ log `type(exc).__name__`).
  - `MockPaymentProvider`: payload thiếu `provider_ref`/`result` không hợp lệ → `ValueError` → router trả 400.
- [x] **§4.6 (Confidential, không lưu card data)** — `raw_payload` chỉ lưu vào cột `JSONB` qua `mark_transaction_success/failed` (không xuất hiện trong log); Sepay = chuyển khoản ngân hàng nên payload không chứa card data; schema không có field card.
- [x] **Audit (§4.5)** — `log.info("PAYMENT_SUCCESS user=%s plan=%s tx=%s amount=%d sub=%s provider=%s", ...)` chỉ log định danh + số tiền; lỗi xác thực log `type(exc).__name__`; KHÔNG có dòng log nào chứa `raw_payload`/`Authorization`/API key/secret (verify bằng cách đọc lại toàn bộ `log.*` trong `payment_service.py`/`sepay_provider.py`).

3. Khởi động FE: `cd speak-buddi && npm run dev`.
4. **Test AC-10-02 end-to-end (mock):**
   - Đăng nhập Student → `/pricing` → "Nâng cấp Pro" → `start_checkout()` tạo `payment_transaction` (`pending`) → redirect `/payment/mock?tx=<id>&ref=MOCK-XXXXXXXX`.
   - Bấm "Giả lập thanh toán thành công" → FE gọi `POST /api/payment/webhook/mock {provider_ref, result: "success"}` → BE: `verify_callback` → map transaction → `activate_subscription` tạo `user_subscription` `active` (`expires_at = NOW() + plan.duration_days`) → `mark_transaction_success` set `status='success'`, `paid_at`, `user_subscription_id`.
   - Query DB: `SELECT status, paid_at, user_subscription_id FROM payment_transaction WHERE id=<tx>` → `success`, `paid_at` not null, `user_subscription_id` set; `SELECT status, expires_at FROM user_subscription WHERE id=<sub_id>` → `active`, `expires_at` đúng `starts_at + duration_days`.
   - Re-login (hoặc gọi `/api/auth/refresh`) → JWT mới có `is_paid=true`; gọi 1 endpoint `require_paid` → 200 thay vì 403.
5. **Test idempotency:** Gọi lại `POST /api/payment/webhook/mock` với cùng `provider_ref` → vẫn trả `{"success": true}`, nhưng `SELECT count(*) FROM user_subscription WHERE user_id=...` không tăng, `payment_transaction` vẫn 1 row `success` (kiểm log có dòng `WEBHOOK_DUPLICATE_SKIP`).
6. **Test nhánh failed (mock):** Bấm "Giả lập thanh toán thất bại" → `POST /api/payment/webhook/mock {result: "failed"}` → `payment_transaction.status='failed'`, `failure_reason='failed'`; KHÔNG tạo `user_subscription`.
7. **Test bảo mật webhook (Sepay — cần `PAYMENT_PROVIDER=sepay` + `SEPAY_WEBHOOK_API_KEY` set):**
   - `POST /api/payment/webhook/sepay` thiếu/sai header `Authorization: Apikey <key>` → 401, transaction không đổi.
   - Header đúng + `transferType="in"` + `code` khớp `provider_transaction_id` → activate; `transferAmount < amount_vnd` → log `WEBHOOK_AMOUNT_MISMATCH`, transaction giữ `pending`.
8. **Test provider không hỗ trợ:** `POST /api/payment/webhook/unknown` → 404 `"Unsupported payment provider"`.
9. **Test §4.5:** Sau các bước trên, kiểm log server — chỉ thấy `PAYMENT_SUCCESS user=.../plan=.../tx=.../amount=.../sub=...`, `WEBHOOK_*` (định danh, không payload); KHÔNG xuất hiện `raw_payload`, `Authorization`, hoặc giá trị `SEPAY_WEBHOOK_API_KEY`.
